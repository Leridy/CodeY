## 引言：为什么需要 Agent Loop？

想象你正在和一个助手对话。你说"帮我查一下天气"，助手查完告诉你结果。你说"那帮我订一张明天去北京的机票"，助手开始帮你操作。

这个过程看似简单，但对于 AI Agent 来说，背后需要一个**持续运行的循环机制**来处理：

1. **理解用户意图** — 调用 LLM 分析需求
2. **决定是否需要工具** — LLM 可能需要调用外部工具（查天气、订机票等）
3. **执行工具并获取结果** — 运行工具，拿到结果
4. **整合结果回复用户** — 把工具结果和 LLM 能力结合起来

这个**持续的对话循环**，就是 Agent Loop。

## 一、什么是 Agent Loop？

### 1.1 简单类比

**Agent Loop 就像一个智能客服的工位**：

```
┌─────────────────────────────────────────────────────────┐
│                    Agent Loop 类比                       │
│                                                         │
│   用户来电 → 客服接听 → 需要查系统？ → 查系统 → 回复用户  │
│      │                                      │          │
│      └──────────── 继续对话 ←───────────────┘          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 角色 | 类比 | 对应组件 |
|------|------|----------|
| 客服大脑 | 理解需求、决定下一步 | LLM 调用 |
| 客服双手 | 操作电脑查信息 | 工具执行 |
| 对话记录 | 之前的沟通内容 | 上下文管理 |
| 实时反馈 | 边查边说 | 流式响应 |

### 1.2 技术定义

Agent Loop 是一个**事件驱动的异步循环**，它：

```
┌─────────────────────────────────────────────────────────┐
│                    Agent Loop 定义                       │
│                                                         │
│  输入：用户消息                                           │
│  输出：AI 响应（可能包含工具调用结果）                      │
│                                                         │
│  核心能力：                                               │
│  ✓ 多轮对话支持                                          │
│  ✓ 工具动态调用                                          │
│  ✓ 流式实时响应                                          │
│  ✓ 上下文记忆                                            │
└─────────────────────────────────────────────────────────┘
```

## 二、核心组件介绍

Agent Loop 由四个核心组件协同工作：

```
┌─────────────────────────────────────────────────────────┐
│                    Agent Loop 架构                       │
│                                                         │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐            │
│  │   LLM   │    │  工具   │    │  上下文  │            │
│  │  调用   │    │  执行   │    │  管理   │            │
│  └────┬────┘    └────┬────┘    └────┬────┘            │
│       │              │              │                  │
│       └──────────────┼──────────────┘                  │
│                      │                                 │
│                      ▼                                 │
│               ┌─────────────┐                          │
│               │  流式响应   │                          │
│               │  (SSE)     │                          │
│               └─────────────┘                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.1 LLM 调用组件

**职责**：与大语言模型通信，获取 AI 的理解和决策。

| 特性 | 说明 |
|------|------|
| 多模型支持 | OpenAI、Claude、本地模型等 |
| 统一接口 | 屏蔽不同模型的 API 差异 |
| 流式传输 | 支持实时接收 LLM 输出 |
| 错误处理 | 自动重试、降级处理 |

**类比**：LLM 调用就像客服的"大脑思考"，负责理解用户意图并决定下一步。

### 2.2 工具执行组件

**职责**：执行 LLM 决定调用的外部工具。

```
┌─────────────────────────────────────────────────────────┐
│                    工具执行流程                           │
│                                                         │
│  LLM 输出：需要调用工具                                   │
│       │                                                 │
│       ▼                                                 │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐            │
│  │ 解析    │───→│ 权限    │───→│ 执行    │            │
│  │ 工具调用 │    │  检查   │    │  工具   │            │
│  └─────────┘    └─────────┘    └─────────┘            │
│                      │              │                  │
│                      │              ▼                  │
│                      │    ┌─────────────┐              │
│                      │    │ 返回结果给  │              │
│                      │    │    LLM     │              │
│                      │    └─────────────┘              │
│                      │                                 │
│                      ▼                                 │
│               ┌─────────────┐                          │
│               │ 权限不足时  │                          │
│               │ 返回错误信息 │                          │
│               └─────────────┘                          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 工具类型 | 示例 | 说明 |
|----------|------|------|
| 文件操作 | read_file, write_file | 读写本地文件 |
| 代码执行 | execute_code | 运行代码片段 |
| 网络请求 | http_request | 调用外部 API |
| 数据库操作 | query_db | 查询数据库 |

**类比**：工具执行就像客服的"双手操作"，负责实际执行任务。

### 2.3 上下文管理组件

**职责**：维护对话历史，让 AI 记住之前的内容。

```
┌─────────────────────────────────────────────────────────┐
│                    上下文管理                             │
│                                                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │ 会话上下文                                       │   │
│  │                                                 │   │
│  │  消息 1: "帮我读取 main.rs"                     │   │
│  │  消息 2: [工具调用] read_file("main.rs")        │   │
│  │  消息 3: [工具结果] "fn main() { ... }"        │   │
│  │  消息 4: "帮我修改一下，加个日志"               │   │
│  │  ...                                            │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  存储方式：                                               │
│  • 内存存储 — 当前会话（快，但断开丢失）                    │
│  • 数据库存储 — 持久化（慢，但可恢复）                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 存储方式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 内存存储 | 速度快 | 断开丢失 | 临时会话 |
| 数据库存储 | 可持久化 | 速度较慢 | 长期会话 |

**类比**：上下文管理就像客服的"对话记录本"，记录之前聊了什么。

### 2.4 流式响应组件

**职责**：实时向用户推送 AI 响应，而不是等全部生成完再返回。

```
┌─────────────────────────────────────────────────────────┐
│                    流式响应 vs 普通响应                    │
│                                                         │
│  普通响应（等待完整结果）：                                │
│  用户 → [等待 10 秒] → 完整响应                          │
│                                                         │
│  流式响应（实时推送）：                                   │
│  用户 → 第 1 个字 → 第 2 个字 → ... → 完整响应           │
│         (0.1秒)     (0.2秒)          (10秒)             │
│                                                         │
│  用户体验：流式响应感觉更快，因为用户能立即看到输出          │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

| 特性 | 说明 |
|------|------|
| 协议 | SSE (Server-Sent Events) |
| 实时性 | 毫秒级推送 |
| 双向性 | 单向（服务器 → 客户端） |
| 断线重连 | 自动重连机制 |

**类比**：流式响应就像客服"边查边说"，而不是查完才告诉你。

## 三、工具调用协议对比

目前主流的工具有两种调用协议：**Function Calling** 和 **Tool Use**。它们的目的相同，但实现方式有差异。

### 3.1 协议对比表

| 特性 | Function Calling | Tool Use |
|------|------------------|----------|
| 主要模型 | OpenAI GPT 系列 | Claude 系列 |
| 工具定义位置 | `functions` 参数 | `tools` 参数 |
| 调用方式 | 单函数调用 | 多工具并行调用 |
| 返回格式 | JSON 对象 | 结构化内容块 |
| 流式支持 | 支持 | 支持 |

### 3.2 流程对比

```
┌─────────────────────────────────────────────────────────┐
│                  Function Calling 流程                   │
│                                                         │
│  用户输入                                               │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │  GPT 模型   │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 函数调用    │ → 执行函数 → 返回结果给 GPT            │
│  └─────────────┘                                        │
│         │                                               │
│         ▼                                               │
│     最终响应                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│                    Tool Use 流程                         │
│                                                         │
│  用户输入                                               │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │ Claude 模型 │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 多工具并行  │ → 同时执行多个工具 → 返回结果           │
│  └─────────────┘                                        │
│         │                                               │
│         ▼                                               │
│     最终响应                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 CodeY 的适配策略

CodeY 采用**适配器模式**，统一处理两种协议：

```
┌─────────────────────────────────────────────────────────┐
│                    适配器架构                             │
│                                                         │
│  ┌─────────────┐                                        │
│  │   Agent     │                                        │
│  │   Loop      │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────────────────────────────────────────┐   │
│  │              统一工具接口                         │   │
│  │         (trait Tool)                            │   │
│  └──────┬─────────────────────────────┬────────────┘   │
│         │                             │                │
│         ▼                             ▼                │
│  ┌─────────────┐              ┌─────────────┐         │
│  │ Function    │              │ Tool Use    │         │
│  │ Calling     │              │ Adapter     │         │
│  │ Adapter     │              │             │         │
│  └─────────────┘              └─────────────┘         │
│         │                             │                │
│         ▼                             ▼                │
│  ┌─────────────┐              ┌─────────────┐         │
│  │ OpenAI API  │              │ Claude API  │         │
│  └─────────────┘              └─────────────┘         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心优势**：
- 工具开发者只需实现一次 `Tool` trait
- 适配器自动转换为不同协议格式
- 新增模型支持只需添加新适配器

## 四、流式响应实现方案

### 4.1 SSE 协议简介

SSE (Server-Sent Events) 是一种**服务器向客户端推送数据的协议**，基于 HTTP。

```
┌─────────────────────────────────────────────────────────┐
│                    SSE 数据格式                           │
│                                                         │
│  data: {"content": "你", "tool_call": null}             │
│                                                         │
│  data: {"content": "好", "tool_call": null}             │
│                                                         │
│  data: {"content": "！", "tool_call": null}             │
│                                                         │
│  data: [DONE]                                           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 StreamManager 工作流程

```
┌─────────────────────────────────────────────────────────┐
│                  StreamManager 流程                      │
│                                                         │
│  ┌─────────────┐                                        │
│  │  start      │ → 创建 channel，返回 receiver          │
│  │  stream     │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │  send       │ → 通过 channel 发送数据块              │
│  │  chunk      │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │  end        │ → 关闭 channel，通知结束               │
│  │  stream     │                                        │
│  └─────────────┘                                        │
│                                                         │
│  数据块 (StreamChunk) 结构：                             │
│  ┌─────────────────────────────────────────────────┐   │
│  │  content: String     // 文本内容                 │   │
│  │  tool_call: Option   // 工具调用（如果有）       │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.3 流式响应的优势

| 对比维度 | 普通响应 | 流式响应 |
|----------|----------|----------|
| 首次响应时间 | 长（等待完整生成） | 短（立即开始推送） |
| 用户体验 | 差（长时间等待） | 好（实时反馈） |
| 资源占用 | 一次性占用 | 渐进式占用 |
| 断线处理 | 丢失全部内容 | 可从断点继续 |
| 实现复杂度 | 简单 | 较复杂 |

## 六、设计决策和权衡

### 6.1 关键决策点

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 语言选择 | Rust | 性能好、内存安全、异步支持好 |
| 异步运行时 | Tokio | Rust 生态最成熟的异步运行时 |
| 流式协议 | SSE | 简单、基于 HTTP、浏览器原生支持 |
| 工具调用 | 双协议支持 | 兼容 OpenAI 和 Claude |
| 上下文存储 | 双模式 | 内存快 + 数据库持久化 |

### 6.2 权衡分析

#### 权衡 1：流式 vs 普通响应

```
┌─────────────────────────────────────────────────────────┐
│                    权衡：流式 vs 普通                     │
│                                                         │
│  选择流式：                                               │
│  ✓ 用户体验好                                           │
│  ✓ 首次响应快                                           │
│  ✗ 实现复杂度高                                         │
│  ✗ 需要处理断线重连                                      │
│                                                         │
│  选择普通：                                               │
│  ✓ 实现简单                                             │
│  ✓ 错误处理容易                                          │
│  ✗ 用户体验差                                           │
│  ✗ 长时间等待                                            │
│                                                         │
│  最终决策：流式响应（用户体验优先）                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 权衡 2：单协议 vs 双协议

```
┌─────────────────────────────────────────────────────────┐
│                  权衡：单协议 vs 双协议                    │
│                                                         │
│  单协议（只支持一种）：                                    │
│  ✓ 实现简单                                             │
│  ✓ 维护成本低                                            │
│  ✗ 只能用一个模型                                        │
│  ✗ 用户选择受限                                          │
│                                                         │
│  双协议（都支持）：                                       │
│  ✓ 用户自由选择模型                                      │
│  ✓ 更好的兼容性                                          │
│  ✗ 实现复杂度高                                         │
│  ✗ 需要适配器层                                          │
│                                                         │
│  最终决策：双协议（灵活性优先）                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

#### 权衡 3：内存存储 vs 数据库存储

```
┌─────────────────────────────────────────────────────────┐
│                权衡：内存 vs 数据库存储                    │
│                                                         │
│  内存存储：                                               │
│  ✓ 速度快（纳秒级）                                      │
│  ✓ 实现简单                                              │
│  ✗ 断开丢失                                              │
│  ✗ 内存占用大                                            │
│                                                         │
│  数据库存储：                                             │
│  ✓ 可持久化                                              │
│  ✓ 可恢复                                                │
│  ✗ 速度慢（毫秒级）                                      │
│  ✗ 实现复杂                                              │
│                                                         │
│  最终决策：两者都支持，用户按需选择                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 七、总结

### 7.1 核心要点回顾

| 组件 | 职责 | 关键特性 |
|------|------|----------|
| LLM 调用 | 理解意图、决策 | 多模型支持、流式传输 |
| 工具执行 | 执行任务 | 权限检查、沙箱隔离 |
| 上下文管理 | 记忆对话 | 双模式存储、自动清理 |
| 流式响应 | 实时反馈 | SSE 协议、断线重连 |

### 7.2 技术亮点

1. **适配器模式** — 统一处理 Function Calling 和 Tool Use
2. **流式传输** — SSE 实现实时响应，提升用户体验
3. **双协议支持** — 兼容主流 LLM 模型
4. **上下文双模式** — 内存快 + 数据库持久化

### 7.3 适用场景

CodeY 的 Agent Loop 设计适用于：

- **AI 编程助手** — 代码生成、文件操作、调试
- **智能客服** — 多轮对话、工具调用
- **自动化工作流** — 任务编排、工具链执行
- **知识问答** — 检索增强生成 (RAG)
