## 一、PathValidator 重命名：解决命名歧义

### 1.1 问题

原来的 `SandboxManager` 名字容易让人误解。在 Phase 2.3 中，`SandboxManager` 是一个 trait，负责操作系统级别的沙箱隔离（macOS Seatbelt、Linux bubblewrap）。而 `permission::sandbox` 模块中的 `SandboxManager` 只做路径校验，两者职责完全不同。

```
┌─────────────────────────────────────────────────────────┐
│                    命名歧义问题                           │
│                                                         │
│  sandbox::SandboxManager (trait)                        │
│       │                                                 │
│       ├── SeatbeltSandboxManager (macOS 沙箱)           │
│       └── BubblewrapSandboxManager (Linux 沙箱)         │
│                                                         │
│  permission::sandbox::SandboxManager ← 容易混淆！       │
│       │                                                 │
│       └── 只做路径校验，跟沙箱无关                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 解决方案

将 `permission::sandbox::SandboxManager` 重命名为 `PathValidator`，职责一目了然：**路径级别的访问控制校验**。

```
┌─────────────────────────────────────────────────────────┐
│                    重命名后                              │
│                                                         │
│  sandbox::SandboxManager (trait)                        │
│       │                                                 │
│       ├── SeatbeltSandboxManager (macOS 沙箱)           │
│       └── BubblewrapSandboxManager (Linux 沙箱)         │
│                                                         │
│  permission::PathValidator ← 清晰！                     │
│       │                                                 │
│       └── 路径校验，职责明确                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.3 PathValidator 核心逻辑

PathValidator 的校验逻辑分三层：

```
┌─────────────────────────────────────────────────────────┐
│                    路径校验流程                           │
│                                                         │
│  输入路径                                                │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │ 规范化路径  │ → 移除 "." 和 ".."，防止路径遍历        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 检查拒绝列表│ → 命中则拒绝（最高优先级）              │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 检查允许列表│ → 命中则允许                            │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 检查工作目录│ → 在工作目录内则允许，否则拒绝          │
│  └─────────────┘                                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

PathValidator 的核心结构体：

```rust
pub struct PathValidator {
    working_directory: PathBuf,
    allowed_paths: Vec<PathBuf>,
    denied_paths: Vec<PathBuf>,
}
```

拒绝优先于允许，工作目录是默认的访问范围。路径规范化通过 `normalize_path()` 函数实现，不依赖文件系统，仅处理路径组件中的 `.` 和 `..`，防止路径遍历攻击。

## 二、FileExecutor：受保护的文件读写

### 2.1 设计思路

FileExecutor 是 Agent 执行文件操作的入口。每次读写都必须经过 PathValidator 校验，确保 Agent 只能访问工作目录范围内的文件。

```
┌─────────────────────────────────────────────────────────┐
│                    FileExecutor 架构                     │
│                                                         │
│  ┌─────────────┐                                        │
│  │   Agent     │                                        │
│  │   Loop      │                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │   Tool      │                                        │
│  │ Orchestrator│                                        │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐    ┌─────────────┐                    │
│  │ FileExecutor│───→│PathValidator│                    │
│  └──────┬──────┘    └─────────────┘                    │
│         │              校验通过？                        │
│         │              ┌────┴────┐                      │
│         │              │         │                      │
│         │              ▼         ▼                      │
│         │             是        否                      │
│         │              │         │                      │
│         ▼              ▼         ▼                      │
│  ┌─────────────┐  ┌─────────┐ ┌─────────┐             │
│  │ 读/写文件   │  │ 执行    │ │ 返回错误 │             │
│  └─────────────┘  └─────────┘ └─────────┘             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 核心方法

| 方法 | 功能 | 返回值 |
|------|------|--------|
| `read(path)` | 读取文件内容 | `{ "content": "...", "bytes": 1234 }` |
| `write(path, content)` | 写入文件（创建或覆盖） | `{ "success": true, "path": "..." }` |

### 2.3 安全特性

- **路径校验**：所有路径操作都通过 PathValidator
- **自动创建目录**：写入时自动创建父目录（`create_dir_all`）
- **路径遍历防护**：`../../etc/passwd` 这类攻击被拦截
- **异步 I/O**：使用 `tokio::fs` 进行非阻塞文件操作

## 三、ShellExecutor：安全的命令执行

### 3.1 设计思路

ShellExecutor 负责执行 Shell 命令，但不是什么命令都能执行。它有三层防护：

```
┌─────────────────────────────────────────────────────────┐
│                    ShellExecutor 三层防护                │
│                                                         │
│  第一层：危险命令模式匹配                                │
│  ┌─────────────────────────────────────────────────┐   │
│  │  rm -rf /    mkfs    dd if=    fork bomb        │   │
│  │  shutdown    reboot    chmod -R 777 /           │   │
│  └─────────────────────────────────────────────────┘   │
│         │                                               │
│         ▼                                               │
│  第二层：工作目录校验                                    │
│  ┌─────────────────────────────────────────────────┐   │
│  │  PathValidator 校验工作目录是否在允许范围内      │   │
│  └─────────────────────────────────────────────────┘   │
│         │                                               │
│         ▼                                               │
│  第三层：执行超时                                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │  默认 30 秒超时，防止命令挂起                    │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 执行流程

```
┌─────────────────────────────────────────────────────────┐
│                    ShellExecutor 执行流程                │
│                                                         │
│  输入命令                                                │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │ 危险命令    │ → 匹命中 → 返回错误                    │
│  │ 模式匹配    │                                        │
│  └──────┬──────┘                                        │
│         │ 未命中                                        │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 工作目录    │ → 不允许 → 返回错误                    │
│  │ 校验        │                                        │
│  └──────┬──────┘                                        │
│         │ 允许                                          │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 执行命令    │ → 超时 → 返回超时错误                  │
│  │ (tokio)     │                                        │
│  └──────┬──────┘                                        │
│         │ 成功                                          │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 返回结果    │                                        │
│  │ stdout/stderr/exit_code                             │
│  └─────────────┘                                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.3 危险命令拦截列表

ShellExecutor 维护一个 `BLOCKED_PATTERNS` 常量，包含已知的危险命令模式：

| 模式 | 风险 |
|------|------|
| `rm -rf /` | 删除根目录 |
| `mkfs` | 格式化磁盘 |
| `dd if=` | 磁盘覆写 |
| `:(){:\|:&};:` | Fork bomb |
| `> /dev/sd` | 覆写块设备 |
| `chmod -R 777 /` | 全局权限开放 |
| `shutdown` / `reboot` / `halt` | 系统关机 |

此外，还拦截试图切换到工作目录外的 `cd` 命令（如 `cd /`、`cd ..`）。

### 3.4 跨平台支持

ShellExecutor 自动适配不同操作系统：

| 平台 | Shell | 命令格式 |
|------|-------|----------|
| Linux/macOS | `/bin/sh` | `sh -c "command"` |
| Windows | `cmd.exe` | `cmd /C "command"` |


### 4.2 工具分类校验

不同类型的工具采用不同的校验策略：

| 工具类型 | 校验内容 | 示例 |
|----------|----------|------|
| 文件工具 | 校验输入参数中的 `path` 字段 | `file/read`, `file/write`, `file/edit` |
| Shell 工具 | 校验工作目录（由 ShellExecutor 执行） | `shell/execute` |
| 其他工具 | 跳过路径校验 | 自定义工具 |

ToolOrchestrator 使用 `with_path_validator()` builder 方法配置路径校验器，未配置时不进行路径校验（向后兼容）。

## 五、Anthropic Tool Use 支持

### 5.1 什么是 Tool Use？

Anthropic 的 Tool Use 与 OpenAI 的 Function Calling 目的相同，但格式不同。Phase 2.5 让 AnthropicProvider 完整支持 Tool Use，使 Claude 模型能够返回结构化的工具调用。

```
┌─────────────────────────────────────────────────────────┐
│                Anthropic Tool Use 响应格式               │
│                                                         │
│  {                                                      │
│    "content": [                                         │
│      {                                                  │
│        "type": "text",                                  │
│        "text": "我来读取文件。"                          │
│      },                                                 │
│      {                                                  │
│        "type": "tool_use",                              │
│        "id": "toolu_abc",                               │
│        "name": "file/read",                             │
│        "input": { "path": "/tmp/test.txt" }             │
│      }                                                  │
│    ],                                                   │
│    "stop_reason": "tool_use"                            │
│  }                                                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 关键变更

| 变更 | 说明 |
|------|------|
| 请求格式 | 添加 `tools` 字段，序列化工具定义为 `AnthropicToolDefinition` |
| 响应解析 | 遍历 `content` 数组，解析 `text` 和 `tool_use` block |
| 工具结果 | 工具结果转换为 `user` 消息 + `tool_result` content block |
| 能力声明 | `supports_function_calling()` 返回 `true` |

### 5.3 工具结果的特殊处理

Anthropic API 要求工具结果以特殊的格式发送：

```
┌─────────────────────────────────────────────────────────┐
│                工具结果消息转换                           │
│                                                         │
│  内部格式：                                              │
│  Message {                                              │
│    role: "tool",                                        │
│    content: "file content here",                        │
│    tool_call_id: "toolu_abc"                            │
│  }                                                      │
│                                                         │
│  转换为 Anthropic 格式：                                 │
│  AnthropicMessage {                                     │
│    role: "user",                                        │
│    content: [                                           │
│      {                                                  │
│        "type": "tool_result",                           │
│        "tool_use_id": "toolu_abc",                      │
│        "content": "file content here"                   │
│      }                                                  │
│    ]                                                    │
│  }                                                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.4 多轮工具调用

Anthropic Tool Use 支持单次响应中包含多个工具调用，Agent Loop 会依次执行并将结果合并回上下文。

```

│                多轮工具调用流程                           │
│                                                         │
│  用户："读取 a.txt 和 b.txt"                             │
│      │                                                  │

│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 合并结果    │ → 添加到上下文                         │
│  └──────┬──────┘                                        │
│         │                                               │
│         ▼                                               │
│  ┌─────────────┐                                        │
│  │ 重新调用 LLM│ → 返回最终响应                         │
│  └─────────────┘                                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 六、流式响应集成

### 6.1 双模式设计

Phase 2.5 实现了流式/非流式双模式，根据配置和场景自动选择。

```
┌─────────────────────────────────────────────────────────┐
│                    流式/非流式双模式                      │
│                                                         │
│  process_message()                                      │
│      │                                                  │
│      ▼                                                  │
│  ┌─────────────┐                                        │
│  │ 检查配置    │                                        │
│  │ enable_streaming?                                   │
│  └──────┬──────┘                                        │
│         │                                               │
│    ┌────┴────┐                                          │
│    │         │                                          │
│    ▼         ▼                                          │
│  true       false                                       │
│    │         │                                          │
│    ▼         ▼                                          │
│  ┌─────────┐ ┌─────────┐                               │
│  │ 有工具？│ │ 非流式  │                               │
│  └────┬────┘ └─────────┘                               │
│       │                                                 │
│  ┌────┴────┐                                            │
│  │         │                                            │
│  ▼         ▼                                            │
│  是       否                                            │
│  │         │                                            │
│  ▼         ▼                                            │
│  非流式   流式                                          │
│  (强制)   (stream_chat)                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.2 关键决策：有工具时强制非流式

这是一个重要的设计决策。当 LLM 响应中包含工具调用时，流式模式无法正确解析工具调用信息（StreamChunk 不包含 tool_call 字段）。因此，当注册了工具时，Agent Loop 会自动回退到非流式模式。

```
┌─────────────────────────────────────────────────────────┐
│                流式模式工具调用回退                       │
│                                                         │
│  问题：                                                  │
│  流式模式下，StreamChunk 只包含文本 delta                │
│  无法传递 tool_use content block                        │
│                                                         │
│  解决方案：                                               │
│  有工具注册时 → 强制使用非流式模式                       │
│  无工具注册时 → 使用流式模式（纯文本对话）               │
│                                                         │
│  ┌─────────────┐    ┌─────────────┐                    │
│  │ 有工具注册  │    │ 无工具注册  │                    │
│  └──────┬──────┘    └──────┬──────┘                    │
│         │                  │                            │
│         ▼                  ▼                            │
│  ┌─────────────┐    ┌─────────────┐                    │
│  │ 非流式模式  │    │ 流式模式    │                    │
│  │ (完整解析)  │    │ (实时推送)  │                    │
│  └─────────────┘    └─────────────┘                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

代码逻辑非常简洁：

```rust
// 有工具定义时强制使用非流式模式，因为流式模式不支持工具调用解析
let use_streaming = self.config.enable_streaming && !has_tools;
```

## 七、代码审查与修复

Phase 2.5 实现完成后，代码审查发现了两个 CRITICAL 级别的问题，均已修复。

### 7.1 CRITICAL-1：AnthropicMessage.content 类型

**问题**：`AnthropicMessage.content` 原来是 `String` 类型，但 Anthropic API 要求工具结果消息的 content 是数组格式。

**影响**：多轮工具调用时，工具结果无法正确发送给 Anthropic API，导致第二轮工具调用失败。

**修复**：将 `content` 类型从 `String` 改为 `serde_json::Value`，支持字符串和数组两种格式。

```
┌─────────────────────────────────────────────────────────┐
│                修复前 vs 修复后                          │
│                                                         │
│  修复前：                                                │
│  AnthropicMessage { content: String }                   │
│  → 工具结果只能是字符串，无法发送 tool_result block      │
│                                                         │
│  修复后：                                                │
│  AnthropicMessage { content: serde_json::Value }        │
│  → 支持字符串（普通消息）和数组（工具结果）              │
│                                                         │
│  普通消息 → content: "Hello"                            │
│  工具结果 → content: [{ type: "tool_result", ... }]     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 7.2 CRITICAL-2：流式模式工具调用回退

**问题**：启用流式模式时，如果 LLM 返回工具调用，StreamChunk 无法携带工具调用信息。

**影响**：工具调用丢失，Agent 无法执行工具，用户只能看到部分响应。

**修复**：当注册了工具时，强制使用非流式模式。通过检查 `tools` 列表是否为空来决定是否回退。
