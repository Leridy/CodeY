# 为什么 Windows 沙箱支持这么难？

> CodeY 技术博客 | 2026-07-05
>
> 在为 CodeY 添加 Windows 平台支持时，我们发现了一个令人意外的事实：Windows 的沙箱机制远比 macOS 和 Linux 复杂。这篇文章记录了我们的研究过程和最终决策。

---

## 一句话总结

macOS 有 Seatbelt，Linux 有 seccomp-bpf，它们都是"一把瑞士军刀"式的方案。Windows？它有 **5 种以上** 互不兼容的沙箱机制，每种都有不同的适用场景、版本要求和 API 设计。这就是 Windows 沙箱支持难的根本原因。

---

## 1. 什么是沙箱？为什么 AI Agent 需要它？

CodeY 是一个 AI Agent 工具，它会执行用户请求的命令。如果没有沙箱，一个恶意 prompt 就可能让 Agent 删除系统文件、窃取数据或挖矿。

**沙箱的核心作用**：给 Agent 一个"安全围栏"，让它只能访问被允许的资源。

```
┌─────────────────────────────────────────────┐
│              Agent 沙箱工作原理               │
│                                             │
│   ┌───────────┐                             │
│   │  用户请求  │                             │
│   └─────┬─────┘                             │
│         ▼                                   │
│   ┌───────────┐    ┌───────────────────┐    │
│   │  AI Agent │───→│      沙箱         │    │
│   │           │    │                   │    │
│   │           │    │  ✅ 允许：项目目录  │    │
│   │           │    │  ✅ 允许：网络请求  │    │
│   │           │    │  ❌ 禁止：系统目录  │    │
│   │           │    │  ❌ 禁止：其他进程  │    │
│   └───────────┘    └───────────────────┘    │
└─────────────────────────────────────────────┘
```

**理想的沙箱应该满足**：

| 需求 | 说明 |
|------|------|
| 轻量级 | 不应显著拖慢命令执行 |
| 安全 | 能真正阻止恶意操作 |
| 灵活 | 可以精细控制权限 |
| 易用 | 开发者不需要复杂配置 |

## 2. 三平台沙箱机制一览

先来看一张总览表，感受一下三平台的差异：

| 特性 | macOS Seatbelt | Linux seccomp-bpf | Windows（多种） |
|------|---------------|-------------------|----------------|
| 方案数量 | 1 种 | 1 种 | 5+ 种 |
| 配置方式 | Scheme 文本文件 | C 代码/宏 | 各不相同 |
| 隔离粒度 | 系统调用级 | 系统调用级 | 取决于方案 |
| 性能开销 | < 1% | < 1% | 1% ~ 15% |
| 版本要求 | macOS 10.5+ | Linux 3.17+ | 各方案不同 |
| 学习曲线 | 中等 | 较高 | 很高 |

### macOS：Seatbelt -- 优雅的统一方案

macOS 用一个叫做 **Seatbelt（沙盒配置文件）** 的机制搞定一切。

```
┌─────────────────────────────────────────┐
│           macOS Seatbelt 工作流          │
│                                         │
│  应用启动                                │
│      │                                  │
│      ▼                                  │
│  加载 .sb 配置文件                       │
│      │                                  │
│      ▼                                  │
│  内核执行系统调用过滤                     │
│      │                                  │
│      ├── 允许的调用 → 正常执行            │
│      └── 禁止的调用 → 返回错误           │
└─────────────────────────────────────────┘
```

**优势**：一个机制覆盖所有场景，从浏览器到 CLI 工具都能用。

### Linux：seccomp-bpf -- 极致的细粒度

Linux 的方案更加底层，它直接过滤每一个系统调用。

```
┌─────────────────────────────────────────┐
│         Linux seccomp-bpf 工作流         │
│                                         │
│  程序启动                                │
│      │                                  │
│      ▼                                  │
│  定义 BPF 过滤规则                       │
│      │                                  │
│      ▼                                  │
│  加载到内核                              │
│      │                                  │
│      ▼                                  │
│  每次系统调用都经过过滤器                 │
│      │                                  │
│      ├── 白名单中的调用 → 放行            │
│      └── 其他调用 → 直接终止进程          │
└─────────────────────────────────────────┘
```

**优势**：极致的控制粒度，Docker、Chrome、Android 都在用。

### Windows：五种机制，五种哲学

这是问题的关键。Windows 没有一个统一的沙箱方案，而是有多个互不相关的机制：

```
┌──────────────────────────────────────────────────────┐
│              Windows 沙箱机制全景图                     │
│                                                      │
│  ┌─────────────────┐  ┌─────────────────┐            │
│  │ Windows Sandbox │  │  AppContainer   │            │
│  │  (完整 VM 隔离)  │  │  (UWP 应用隔离)  │            │
│  │  需要 Pro/企业版  │  │  需要 Windows 8+ │            │
│  │  启动慢，开销大   │  │  配置复杂        │            │
│  └─────────────────┘  └─────────────────┘            │
│                                                      │
│  ┌─────────────────┐  ┌─────────────────┐            │
│  │  Job Objects    │  │Restricted Tokens│            │
│  │  (资源限制)      │  │  (权限限制)      │            │
│  │  隔离强度低      │  │  配置非常复杂    │            │
│  │  适合资源管控    │  │  需深入理解安全模型│            │
│  └─────────────────┘  └─────────────────┘            │
│                                                      │
│  ┌─────────────────┐                                 │
│  │Mandatory        │                                 │
│  │Integrity Control│                                 │
│  │  (完整性级别)    │                                 │
│  │  基础防护层      │                                 │
│  └─────────────────┘                                 │
└──────────────────────────────────────────────────────┘
```

## 3. 逐个拆解：Windows 五种沙箱机制

### 3.1 Windows Sandbox -- 最安全但最重

这是微软官方的"一次性桌面"方案，基于 Hyper-V 虚拟化。

**类比**：就像你每次用电脑都开一台全新的虚拟机，用完就扔掉。

| 优点 | 缺点 |
|------|------|
| 隔离级别最高（Hyper-V 级） | 只支持 Pro/企业版 |
| 零配置开箱即用 | 启动要好几秒 |
| 关闭即销毁，不留痕迹 | 内存占用 100MB+ |
| 与桌面无缝集成 | 不适合 CLI 工具轻量级需求 |

**问题**：CodeY 是一个 CLI 工具，用户希望 `codey run` 秒级响应，不能每次等几秒启动虚拟机。

### 3.2 AppContainer -- 最灵活但最复杂

这是 Windows 8 引入的 UWP 应用隔离机制，理论上最适合 CLI 工具。

**类比**：就像 Android 的权限系统 -- 应用需要在清单里声明"我要访问网络""我要访问文件"。

| 隔离维度 | 默认行为 | 可配置 |
|---------|---------|--------|
| 文件系统 | 仅访问包目录 | 通过能力声明扩展 |
| 网络 | 仅本地回环 | 需声明 `internetClient` |
| 注册表 | 仅包注册表 | 不可扩展 |
| 进程 | 完全隔离 | 不可跨容器访问 |

**问题**：配置极其复杂，需要调用大量 Win32 API，而且不同 Windows 版本行为不一致。

### 3.3 Job Objects -- 最轻量但最弱

这是 Windows 内核的进程分组机制，主要用于资源限制。

**类比**：就像 Linux 的 cgroups -- 给一组进程设上限（最多 1 个进程、最多 100MB 内存）。

| 优点 | 缺点 |
|------|------|
| 开销可忽略不计 | 不是真正的安全边界 |
| API 简单易用 | 只能限制资源，不能限制权限 |
| 所有 Windows 版本都支持 | 无法防御恶意代码 |

**问题**：它只能"限流"，不能"拦截"。恶意代码仍然可以读取敏感文件。

### 3.4 Restricted Tokens -- 最精细但最难用

通过创建受限的 Windows 安全令牌来限制进程权限。

**类比**：就像给进程发一张"有限权限的员工卡" -- 某些楼层去不了，某些门开不了。

**问题**：需要深入理解 Windows 安全模型（SID、DACL、完整性级别），开发者门槛极高。

### 3.5 Mandatory Integrity Control -- 基础防护

Windows Vista 引入的强制完整性控制，是所有其他机制的基础层。

**核心规则**：低完整性级别的进程不能写入高完整性级别的对象。这是一道"地板"级别的防线。

## 4. 三平台深度对比

### 4.1 性能开销对比

这是 CodeY 最关心的指标 -- 用户不能等沙箱启动半天。

| 方案 | 启动延迟 | 内存开销 | CPU 开销 | 隔离强度 |
|------|---------|---------|---------|---------|
| macOS Seatbelt | 极低 | 极低 | < 1% | 中 |
| Linux seccomp-bpf | 极低 | 极低 | < 1% | 高 |
| Windows AppContainer | 低 | 低 | < 2% | 中 |
| Windows Job Objects | 可忽略 | 可忽略 | < 1% | 低 |
| **Windows Sandbox** | **高（秒级）** | **高（100MB+）** | **5-15%** | **非常高** |
| **Docker + WSL2** | **高（秒级）** | **高（1GB+）** | **5-10%** | **非常高** |
| Sandboxie | 低 | 低 | 2-5% | 中 |

**关键发现**：

```
┌─────────────────────────────────────────────────┐
│          性能 vs 安全 的权衡图谱                  │
│                                                 │
│  安全性 ▲                                       │
│    高   │  ● Windows Sandbox                    │
│         │  ● Docker + WSL2                      │
│         │                                       │
│    中   │       ● AppContainer                  │
│         │       ● Sandboxie                     │
│         │  ● seccomp-bpf                        │
│         │  ● Seatbelt                           │
│    低   │  ● Job Objects                        │
│         └────────────────────────────────→ 性能  │
│              低      中       高                 │
└─────────────────────────────────────────────────┘
```

**结论**：macOS 和 Linux 的方案在性能和安全之间取得了最佳平衡。Windows 要么太重（Sandbox、Docker），要么太弱（Job Objects），要么太复杂（AppContainer）。

### 4.2 开发者体验对比

| 维度 | macOS | Linux | Windows |
|------|-------|-------|---------|
| 配置文件格式 | Scheme `.sb` 文件 | C 宏/BPF 字节码 | XML/Win32 API/各不相同 |
| 学习曲线 | 中等 | 较高 | **非常高** |
| 文档质量 | 良好 | 优秀 | 碎片化 |
| 社区支持 | 成熟 | 非常成熟 | 中等 |
| 调试难度 | 低 | 中 | **高** |
| 跨版本一致性 | 高 | 高 | **低** |

### 4.3 版本碎片化问题

这是 Windows 特有的痛点：

| 方案 | 最低版本要求 | 备注 |
|------|------------|------|
| Windows Sandbox | Windows 10 Pro 1903 | Home 版不支持 |
| AppContainer | Windows 8 | 行为因版本而异 |
| Job Objects | Windows XP | 所有版本支持 |
| Restricted Tokens | Windows 2000 | 所有版本支持 |
| WSL2 | Windows 10 1903 | 需要启用虚拟化 |

**对比**：macOS 和 Linux 几乎不存在这个问题。macOS 10.5 以后 Seatbelt 就是标准；Linux 内核 3.17+ 都支持 seccomp-bpf。

## 5. 我们的决策：MVP 不支持原生 Windows

经过深入研究，我们做出了一个艰难但合理的决定：

```
┌─────────────────────────────────────────────────┐
│              CodeY MVP 平台支持策略              │
│                                                 │
│  ┌─────────┐  ┌─────────┐  ┌──────────────┐    │
│  │ macOS   │  │ Linux   │  │   Windows    │    │
│  │         │  │         │  │              │    │
│  │ ✅ 原生  │  │ ✅ 原生  │  │ ⚠️ 仅 WSL2  │    │
│  │Seatbelt │  │seccomp  │  │  (Linux 环境) │    │
│  └─────────┘  └─────────┘  └──────────────┘    │
│                                                 │
│  理由：复用 Linux 沙箱实现，一套代码覆盖三平台     │
└─────────────────────────────────────────────────┘
```

### 为什么这样做？

| 考量因素 | 支持原生 Windows | 仅支持 WSL2 |
|---------|-----------------|------------|
| 开发成本 | 需要实现 5 种机制的适配 | 复用现有 Linux 代码 |
| 维护负担 | 三套独立代码 | 一套代码 |
| 测试矩阵 | Windows 7/8/10/11 x Home/Pro/Enterprise | WSL2 环境 |
| 用户覆盖 | 更广 | 覆盖大部分 Windows 开发者 |
| 上线速度 | 慢 | 快 |

### WSL2 的优势

WSL2 运行真实的 Linux 内核，这意味着：

- CodeY 的 Linux 沙箱代码可以直接复用
- 开发者体验与 macOS/Linux 一致
- Docker 生态无缝集成
- GPU 直通支持（未来 ML/AI 工作负载）

### 用户如何使用？

Windows 用户只需安装 WSL2，然后在 WSL2 环境中使用 CodeY：

```
┌─────────────────────────────────────────┐
│         Windows 用户使用流程             │
│                                         │
│  ┌─────────┐                            │
│  │ Windows │                            │
│  │  桌面   │                            │
│  └────┬────┘                            │
│       ▼                                 │
│  ┌─────────┐                            │
│  │  WSL2   │ ← 真实 Linux 内核          │
│  │ Ubuntu  │                            │
│  └────┬────┘                            │
│       ▼                                 │
│  ┌─────────┐                            │
│  │  CodeY  │ ← 复用 Linux 沙箱实现      │
│  │  Agent  │                            │
│  └─────────┘                            │
└─────────────────────────────────────────┘
```

## 6. 长期路线图

MVP 之后，我们计划分阶段支持原生 Windows：

```
┌─────────────────────────────────────────────────────┐
│              Windows 支持路线图                       │
│                                                     │
│  Phase 1 (MVP)          Phase 2 (v2.0)              │
│  ┌──────────────┐       ┌──────────────────┐        │
│  │   WSL2 支持   │──────→│ AppContainer +   │        │
│  │ 复用 Linux 代码│       │ Job Objects 组合  │        │
│  └──────────────┘       └────────┬─────────┘        │
│                                  │                   │
│                                  ▼                   │
│                          Phase 3 (v3.0)              │
│                          ┌──────────────────┐        │
│                          │ 完整 Windows 支持  │        │
│                          │ 多种隔离级别可选    │        │
│                          └──────────────────┘        │
│                                                     │
│  参考：Chromium 的 Windows 沙箱实现                    │
└─────────────────────────────────────────────────────┘
```

| 阶段 | 方案 | 隔离强度 | 适用场景 |
|------|------|---------|---------|
| Phase 1 | WSL2（Linux 沙箱） | 高 | 所有 Windows 开发者 |
| Phase 2 | AppContainer + Job Objects | 中-高 | 需要原生体验的用户 |
| Phase 3 | 多方案可选 | 可选 | 根据安全需求选择 |

### Phase 2 的技术方向

Phase 2 将参考 Chromium 的 Windows 沙箱设计，采用 **AppContainer + Job Objects 组合**：

```
┌─────────────────────────────────────────┐
│      Phase 2 沙箱架构（概念）            │
│                                         │
│  ┌─────────────────────────────────┐    │
│  │         AppContainer            │    │
│  │   （能力控制：文件、网络、进程）  │    │
│  │                                 │    │
│  │   ┌─────────────────────────┐   │    │
│  │   │     Job Objects         │   │    │
│  │   │  （资源限制：CPU、内存）  │   │    │
│  │   │                         │   │    │
│  │   │   ┌─────────────────┐   │   │    │
│  │   │   │  Agent 进程     │   │   │    │
│  │   │   └─────────────────┘   │   │    │
│  │   └─────────────────────────┘   │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

这种组合提供了：
- AppContainer 的能力控制（哪些资源可以访问）
- Job Objects 的资源限制（CPU、内存上限）
- 两者叠加的多层防护

### 降级策略

不是所有 Windows 环境都支持所有方案，所以我们需要智能降级：

```
┌─────────────────────────────────────────┐
│          沙箱降级策略                     │
│                                         │
│  开始                                   │
│   │                                     │
│   ▼                                     │
│  检测：是否支持 AppContainer？            │
│   │                                     │
│   ├── 是 → 使用 AppContainer + Job      │
│   │        Objects 组合                  │
│   │                                     │
│   └── 否 → 检测：是否支持 Job Objects？  │
│              │                          │
│              ├── 是 → 仅使用 Job Objects │
│              │        （资源限制模式）     │
│              │                          │
│              └── 否 → 无沙箱模式         │
│                       （受限功能警告）     │
└─────────────────────────────────────────┘
```

## 7. 给其他开发者的建议

如果你也在做跨平台的沙箱隔离，这是我们的经验总结：

| 建议 | 说明 |
|------|------|
| 优先 macOS/Linux | 这两个平台有成熟的统一方案 |
| Windows 用 WSL2 兜底 | 大部分开发者已经装了，复用 Linux 代码 |
| 参考 Chromium | 它是最权威的 Windows 沙箱实践 |
| 不要追求完美 | 先用 Job Objects 做基础防护，再逐步增强 |
| 做好降级 | 不同 Windows 版本差异大，必须有 fallback |

---

## 总结

Windows 沙箱支持难，不是因为技术不行，而是因为 **碎片化**。

- macOS 和 Linux 各有一个统一的、经过实战检验的沙箱机制
- Windows 有 5 种以上互不兼容的机制，每种都有版本限制和适用场景
- 对于 CodeY 这样的 CLI 工具，没有一个 Windows 方案能像 macOS Seatbelt 或 Linux seccomp-bpf 那样"开箱即用"

我们的解法是务实的：**MVP 阶段通过 WSL2 复用 Linux 实现，长期逐步支持原生 Windows**。这不是妥协，而是基于技术分析的理性选择。
