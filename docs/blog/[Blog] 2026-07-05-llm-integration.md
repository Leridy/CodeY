## 1. 引言 - 为什么需要统一的 LLM 集成？

### 一个常见的痛点

你是否有过这样的经历？

- 用 OpenAI 的 API 写了一套代码
- 想切换到 DeepSeek 节省成本，发现 API 格式不一样
- 想试试本地 Ollama 模型，又要重写一遍调用逻辑
- 团队里有人用 Anthropic，有人用通义千问，代码里到处是 `if-else`

```
┌─────────────────────────────────────────────────────────┐
│                    痛点：碎片化集成                       │
```

## 2. 设计思路 - cc-switch 风格的提供商管理

### 灵感来源：Claude Code 的 Provider Switch

在设计 CodeY 的 LLM 集成时，我们参考了 Claude Code 的 `cc-switch` 工具 -- 它让用户可以在不同 Claude 模型之间快速切换。我们把这个思路扩展到了**所有主流 LLM 提供商**。

### 设计目标

| 目标 | 说明 |
|------|------|
| 统一接口 | 所有提供商实现同一个 Trait |
| 热切换 | 运行时切换提供商，无需重启 |
| 配置持久化 | SQLite 存储，原子写入 |
| 50+ 预设 | 内置主流提供商，开箱即用 |
| 流式优先 | 原生支持 SSE 流式输出 |
| 安全存储 | API Key 通过环境变量读取 |

### 类比：餐厅点菜

把 LLM 集成想象成一家餐厅的后厨系统：

- **菜单**（`LlmProvider` Trait）：定义了所有菜品的标准做法
- **厨师**（具体 Provider）：每个厨师擅长不同的菜系
- **前台**（`ProviderRegistry`）：负责把订单分配给正确的厨师
- **菜谱库**（SQLite）：存储所有厨师的信息和拿手菜

```
┌─────────────────────────────────────────────────────────┐
│                   餐厅后厨系统                           │
│                                                         │
│  客户点菜                                                │
│     │                                                   │
│     ▼                                                   │
│  ┌──────────┐     ┌──────────────┐                      │
│  │  前台    │────→│   菜谱库     │                      │
│  │Registry  │     │   SQLite     │                      │
│  └────┬─────┘     └──────────────┘                      │
│       │                                                 │
│       ▼                                                 │
│  ┌──────────────────────────────┐                       │
│  │         菜单 (Trait)         │                       │
│  │   chat() / stream_chat()    │                       │
│  └──────────┬───────────────────┘                       │
│             │                                           │
│    ┌────────┼────────┬──────────┐                       │
│    ▼        ▼        ▼          ▼                       │
│ ┌──────┐┌───────┐┌────────┐┌────────┐                  │
│ │粤菜  ││川菜   ││西餐    ││日料    │                  │
│ │OpenAI││Claude ││DeepSeek││Ollama │                  │
│ └──────┘└───────┘└────────┘└────────┘                  │
└─────────────────────────────────────────────────────────┘
```

## 3. 架构设计 - Provider Trait + Registry + SQLite

### 三层架构

CodeY 的 LLM 集成采用三层架构，每层职责清晰：

```
┌─────────────────────────────────────────────────────────┐
│                   LLM 集成三层架构                       │
│                                                         │
│  ┌─────────────────────────────────────────────┐        │
│  │              应用层 (Agent Core)             │        │
│  │                                             │        │
│  │  agent.chat()  agent.stream_chat()          │        │
│  └─────────────────┬───────────────────────────┘        │
│                    │                                    │
│  ┌─────────────────▼───────────────────────────┐        │
│  │              管理层 (Registry)               │        │
│  │                                             │        │
│  │  register()  get()  set_active()  list()    │        │
│  └─────────────────┬───────────────────────────┘        │
│                    │                                    │
│  ┌─────────────────▼───────────────────────────┐        │
│  │              存储层 (SQLite)                 │        │
│  │                                             │        │
│  │  providers 表  models 表  config 表          │        │
│  └─────────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────────┘
```

### 核心组件

#### 3.1 LlmProvider Trait -- 统一接口

`LlmProvider` 是整个集成层的核心 Trait，定义了所有提供商必须实现的能力：

```
┌─────────────────────────────────────────────────────────┐
│                LlmProvider Trait                         │
│                                                         │
│  ┌───────────────────────────────────────┐               │
│  │  fn id()          → 提供商标识        │               │
│  │  fn name()        → 显示名称          │               │
│  │  fn models()      → 可用模型列表      │               │
│  │  fn chat()        → 同步对话          │               │
│  │  fn stream_chat() → 流式对话          │               │
│  │  fn supports_streaming()  → 流式能力  │               │
│  │  fn supports_function_calling()       │               │
│  └───────────────────────────────────────┘               │
│                                                         │
│  关键设计：                                              │
│  - async_trait：异步方法支持                              │
│  - Send + Sync：线程安全                                 │
│  - clone_box()：支持动态分发的克隆                        │
└─────────────────────────────────────────────────────────┘
```

**为什么用 Trait 而不是泛型？**

因为提供商是在运行时从数据库加载的，编译期无法确定具体类型。Trait Object (`Box<dyn LlmProvider>`) 让我们可以在运行时动态切换提供商。

#### 3.2 ProviderRegistry -- 提供商管理器

Registry 是提供商的"管家"，负责注册、查找、切换：

```
┌─────────────────────────────────────────────────────────┐
│                ProviderRegistry                          │
│                                                         │
│  ┌───────────────────────────────────────┐               │
│  │  providers: HashMap<String, Provider> │               │
│  │  active_provider: Option<String>      │               │
│  └───────────────────────────────────────┘               │
│                                                         │
│  操作流程：                                              │
│                                                         │
│  register("openai", provider)                           │
│       │                                                 │
│       ▼                                                 │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│  │ openai  │    │anthropic│    │ deepseek│             │
│  └─────────┘    └─────────┘    └─────────┘             │
│       ▲                                                 │
│       │                                                 │
│  set_active("openai")                                   │
│       │                                                 │
│       ▼                                                 │
│  active() → 返回当前活跃的 Provider                      │
└─────────────────────────────────────────────────────────┘
```

**并发安全**：Registry 使用 `Arc<RwLock<...>>` 包装，支持多线程并发读写。读操作（`get`、`list`）不阻塞，写操作（`register`、`remove`）互斥。

#### 3.3 SQLite 持久化 -- 配置不丢失

SQLite 存储所有提供商配置，三张表各司其职：

```
┌─────────────────────────────────────────────────────────┐
│                   SQLite 数据库结构                      │
│                                                         │
│  ┌──────────────────────────────────────┐               │
│  │          providers 表                │               │
│  │  id | name | base_url | api_key_env  │               │
│  │  chat_endpoint | default_model       │               │
│  └──────────────────────────────────────┘               │
│           │                                             │
│           │ 1:N                                         │
│           ▼                                             │
│  ┌──────────────────────────────────────┐               │
│  │          models 表                   │               │
│  │  id | provider_id | name             │               │
│  │  context_window | max_output_tokens  │               │
│  └──────────────────────────────────────┘               │
│                                                         │
│  ┌──────────────────────────────────────┐               │
│  │          config 表                   │               │
│  │  key | value | updated_at            │               │
│  │  例：active_provider = "openai"      │               │
│  └──────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────┘
```

**为什么选 SQLite 而不是 JSON 文件？**

| 对比项 | JSON 文件 | SQLite |
|--------|-----------|--------|
| 原子写入 | 需要手动实现 | 原生支持 |
| 并发读写 | 需要文件锁 | WAL 模式 |
| 查询能力 | 全量加载 | SQL 查询 |
| 数据完整性 | 无保障 | 事务 + 外键 |
| 文件损坏风险 | 高（写入中断） | 低（WAL） |

### 5.3 添加自定义提供商

想接入新的 LLM 提供商？只需三步：

```
┌─────────────────────────────────────────────────────────┐
│                 添加自定义提供商                          │
│                                                         │
│  步骤 1：创建 Provider 实现                              │
│  ┌──────────────────────────────────────┐               │
│  │ struct MyProvider {                  │               │
│  │   api_key: String,                   │               │
│  │   base_url: String,                  │               │
│  │   client: Client,                    │               │
│  │ }                                    │               │
│  │                                      │               │
│  │ impl LlmProvider for MyProvider {    │               │
│  │   // 实现所有必需方法                 │               │
│  │ }                                    │               │
│  └──────────────────────────────────────┘               │
│                                                         │
│  步骤 2：注册到 Loader                                   │
│  ┌──────────────────────────────────────┐               │
│  │ match config.id.as_str() {           │               │
│  │   "my_provider" => Some(Box::new(    │               │
│  │     MyProvider::new(api_key)         │               │
│  │   )),                                │               │
│  │   _ => None                          │               │
│  │ }                                    │               │
│  └──────────────────────────────────────┘               │
│                                                         │
│  步骤 3：添加数据库配置                                   │
│  ┌──────────────────────────────────────┐               │
│  │ INSERT INTO providers VALUES (       │               │
│  │   'my_provider', 'My Provider', ...  │               │
│  │ );                                   │               │
│  └──────────────────────────────────────┘               │
│                                                         │
│  完成！现在可以通过 Registry 使用新提供商                  │
└─────────────────────────────────────────────────────────┘
```

## 7. 总结 - 学到了什么

### 7.1 设计经验

```
┌─────────────────────────────────────────────────────────┐
│                   设计经验总结                            │
│                                                         │
│  1. Trait Object > 泛型                                  │
│     ┌──────────────────────────────────────┐            │
│     │ 运行时加载的提供商，必须用 Trait Object│            │
│     │ Box<dyn LlmProvider> 灵活但有开销     │            │
│     │ clone_box() 模式解决克隆问题          │            │
│     └──────────────────────────────────────┘            │
│                                                         │
│  2. SQLite > JSON 文件                                   │
│     ┌──────────────────────────────────────┐            │
│     │ 原子写入防止配置损坏                  │            │
│     │ WAL 模式支持并发读写                  │            │
│     │ 外键约束保证数据完整性                │            │
│     └──────────────────────────────────────┘            │
│                                                         │
│  3. 统一格式 + 适配器模式                                │
│     ┌──────────────────────────────────────┐            │
│     │ 内部统一 Message/ChatRequest 格式     │            │
│     │ 每个 Provider 负责格式转换            │            │
│     │ 上层代码完全不关心底层实现            │            │
│     └──────────────────────────────────────┘            │
│                                                         │
│  4. 流式优先                                             │
│     ┌──────────────────────────────────────┐            │
│     │ LLM 应用的核心体验是流式输出          │            │
│     │ mpsc::channel 实现异步流式传输        │            │
│     │ tokio::spawn 处理后台流式任务         │            │
│     └──────────────────────────────────────┘            │
│                                                         │
│  5. 环境变量存储 API Key                                 │
│     ┌──────────────────────────────────────┐            │
│     │ 永远不要硬编码 API Key                │            │
│     │ SQLite 只存环境变量名，不存实际值      │            │
│     │ 启动时从环境变量读取                  │            │
│     └──────────────────────────────────────┘            │
└─────────────────────────────────────────────────────────┘
```
