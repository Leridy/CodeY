# [Blog] Phase 3 前端 UI 技术规划

> 日期：2026-07-05
> 阶段：技术分析
> 作者：CodeY Team

## 引言

CodeY 的后端核心已经完成（Phase 1-2），包括 LLM 集成、权限系统、沙箱系统和 Agent Loop。现在是时候构建用户真正使用的界面了。

Phase 3 的目标是将后端能力转化为用户可见的产品 -- 一个 IDE 风格的 AI 编程助手界面。本文记录了 Phase 3 的技术分析、选型决策和拆分方案。

### 1.3 多项目多会话

这是一个关键架构决策。用户可能同时处理多个项目，每个项目有多个独立会话：

```
┌─────────────────────────────────────────────────────────────┐
│                    多项目多会话模型                           │
│                                                             │
│  Project A (~/code/frontend)                                │
│  ├── Session 1: "重构登录页面"                              │
│  ├── Session 2: "修复 CSS bug"                              │
│  └── Session 3: "添加暗色主题"                              │
│                                                             │
│  Project B (~/code/backend)                                 │
│  ├── Session 1: "实现用户 API"                              │
│  └── Session 2: "优化数据库查询"                            │
│                                                             │
│  关系: 项目 1:N 会话                                        │
│  隔离: 每个会话独立的 Agent 上下文                           │
└─────────────────────────────────────────────────────────────┘
```

**Web 端项目目录方案**：

| 方案 | 说明 | 优势 | 劣势 |
|------|------|------|------|
| File System Access API | 浏览器原生文件系统访问 | 真实文件操作、无需服务器 | 仅 Chromium 支持 |
| 服务器端文件管理 | 通过 API 读写文件 | 兼容性好 | 需要常驻服务器 |
| 虚拟文件系统 | 内存中的文件系统 | 简单、安全 | 无法持久化 |

**决策**: Web 端使用 File System Access API（Chromium 系浏览器），Desktop 端通过 Tauri 直接访问文件系统。

## 2. 技术选型

### 2.1 状态管理

前端需要管理多种状态：布局、聊天、编辑器、主题、项目、会话。

**候选方案对比**：

```
┌─────────────────────────────────────────────────────────────┐
│                    状态管理方案对比                           │
│                                                             │
│  Redux Toolkit          Zustand             Jotai           │
│  ┌──────────────┐      ┌──────────────┐   ┌──────────────┐│
│  │ 单一 Store    │      │ 独立 Store    │   │ 原子化状态    ││
│  │ 严格中间件    │      │ 灵活组合      │   │ 细粒度订阅    ││
│  │ 样板代码多    │      │ 代码量少      │   │ 学习曲线高    ││
│  │ 生态完善      │      │ TypeScript 好 │   │ 适合简单状态  ││
│  └──────────────┘      └──────────────┘   └──────────────┘│
│         ↓                      ↓                 ↓         │
│       不采用                  采用              不采用       │
└─────────────────────────────────────────────────────────────┘
```

**决策: Zustand**

理由：
- 项目已有 Zustand 基础（appStore.ts）
- 代码量少，样板代码几乎为零
- TypeScript 支持好
- persist 中间件支持 localStorage 持久化
- 可以按领域拆分多个 Store（layoutStore、chatStore、editorStore）

### 2.2 UI 组件库

**候选方案对比**：

| 方案 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| Radix UI | 无障碍、无样式、可定制 | 需要自行样式 | **采用（基础层）** |
| shadcn/ui | 开箱即用、美观 | 依赖 Tailwind、样式固定 | 参考但不直接采用 |
| Headless UI | Tailwind 官方 | 组件少 | 不采用 |
| 自研组件 | 完全可控 | 工作量大 | 部分采用 |

**决策: Radix UI + 自定义样式**

理由：
- Radix UI 提供无障碍基础（键盘导航、屏幕阅读器）
- 无样式设计与 codey-frontend-style skill 完美配合
- 自定义样式确保 IDE 风格一致性
- Tooltip、Popover、Dialog 等基础组件直接使用 Radix

### 2.3 样式方案

**决策: Tailwind CSS**

理由：
- 项目已配置 Tailwind
- 工具类优先，减少 CSS 文件
- 与 codey-frontend-style skill 的 CSS 变量系统兼容
- 响应式工具类支持断点适配

### 2.4 动画方案

**决策: Framer Motion**

理由：
- 项目已有 Framer Motion 基础
- 声明式动画 API，易于维护
- 支持布局动画（layout animation）
- AnimatePresence 支持组件卸载动画
- 性能优秀，使用 GPU 加速

## 3. Phase 3 拆分方案

Phase 3 功能较多，需要合理拆分为可交付的子阶段。

### 3.1 拆分原则

- 每个子阶段独立可交付
- 优先级高的功能先做
- 后续阶段依赖前序阶段的基础设施

## 5. 风险与挑战

### 5.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| CSS Grid 拖拽性能 | 面板调整卡顿 | 使用 requestAnimationFrame 优化 |
| localStorage 容量限制 | 复杂布局无法保存 | 压缩存储数据，限制自定义预设数量 |
| File System Access API 兼容性 | 非 Chromium 浏览器不可用 | 提供服务器端 fallback |
| Framer Motion 包体积 | 首屏加载慢 | 按需导入，tree-shaking |

### 5.2 设计挑战

| 挑战 | 说明 | 解决方案 |
|------|------|----------|
| 面板布局灵活性 vs 一致性 | 用户自由调整可能破坏设计美感 | 提供预设 + 约束规则 |
| 多会话上下文切换 | 切换会话时状态恢复 | 每个会话独立 Store slice |
| 移动端适配 | IDE 布局在小屏幕上体验差 | 移动端简化为单面板 + Tab |

## 总结

Phase 3 是 CodeY 从"后端能力"到"用户体验"的关键转折。通过合理的拆分和技术选型，我们可以逐步构建一个功能完善、体验流畅的 IDE 风格界面。

**关键决策回顾**：

| 决策 | 选择 | 理由 |
|------|------|------|
| 状态管理 | Zustand | 轻量、已有基础、TypeScript 好 |
| UI 组件 | Radix UI + 自定义 | 无障碍基础 + IDE 风格一致性 |
| 样式方案 | Tailwind CSS | 工具类优先、响应式支持 |
| 动画方案 | Framer Motion | 声明式、布局动画、性能好 |
| 拖拽方案 | CSS Grid + 自定义 | 轻量、完全可控 |
| 多项目方案 | File System Access API | 浏览器原生、无需服务器 |

Phase 3.1 已经规划完成，Spec 文档已生成。接下来就是动手实现。
