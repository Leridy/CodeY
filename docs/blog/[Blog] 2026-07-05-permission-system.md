## 1. 引言 - 为什么需要权限系统？

想象一下这个场景：

你让 AI Agent 帮你重构一个项目。Agent 二话不说，直接 `rm -rf` 了你的 `node_modules`，然后重新 `npm install`。结果呢？依赖版本全变了，项目跑不起来了。

或者更糟：Agent 觉得你的 `.env` 文件"不够安全"，自作主张帮你"加密"了，然后你的数据库连接全断了。

**这就是为什么我们需要权限系统。**

AI Agent 不是人类开发者。它没有"常识"，不知道哪些目录不能碰，哪些文件不能改。它只会按照指令执行，但执行的方式可能让你崩溃。

我们需要一套系统，让 Agent 知道：

```
┌─────────────────────────────────────────────────────┐
│                   Agent 的边界                       │
│                                                     │
│  "我可以读这个文件吗？"                              │
│  "我可以写这个目录吗？"                              │
│  "我可以执行这个命令吗？"                            │
│  "我可以访问网络吗？"                                │
│                                                     │
│  每一步都需要明确的答案。                            │
└─────────────────────────────────────────────────────┘
```

## 2. 设计思路 - 细粒度权限 + 规则引擎

### 2.1 为什么不用简单的"允许/拒绝"？

很多工具用的是二元权限：要么全开，要么全关。但这在实际开发中根本不够用。

**类比**：这就像你家的门锁。你不会只有一把钥匙，对吧？

- 大门钥匙 -- 只能开门
- 卧室钥匙 -- 可以进卧室
- 保险箱钥匙 -- 可以开保险箱

你不会把保险箱钥匙给保姆，但你可能会给她大门钥匙。

权限系统也是一样。Agent 需要的是**按需分配的最小权限**。

### 2.2 三层架构

我们设计了三层权限架构：

```
┌─────────────────────────────────────────────────────┐
│                 权限系统架构                          │
│                                                     │
│  ┌───────────────────────────────────────────────┐  │
│  │  第一层：7 级权限模型                          │  │
│  │  定义 Agent 能做什么（宏观权限）               │  │
│  └───────────────────────────────────────────────┘  │
│                      │                              │
│                      ▼                              │
│  ┌───────────────────────────────────────────────┐  │
│  │  第二层：规则引擎 (DSL)                       │  │
│  │  定义 Agent 对哪些资源能做什么（精细控制）     │  │
│  └───────────────────────────────────────────────┘  │
│                      │                              │
│                      ▼                              │
│  ┌───────────────────────────────────────────────┐  │
│  │  第三层：OS 级沙箱                            │  │
│  │  在操作系统层面限制 Agent 的能力（最后防线）   │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

这三层各司其职：

| 层级 | 职责 | 类比 |
|------|------|------|
| 7 级权限模型 | 定义 Agent 的权限等级 | 公司的职级体系 |
| 规则引擎 | 针对具体资源的精细控制 | 部门的具体规章制度 |
| OS 级沙箱 | 操作系统层面的硬隔离 | 公司的物理门禁系统 |

## 3. 架构设计 - 7 级权限模型

### 3.1 权限等级

我们把权限分成了 7 个等级，从最保守到最开放：

```
Level 7: FullAccess        --- 完全访问权限
Level 6: Network           --- 网络访问
Level 5: ShellWrite        --- Shell 写入命令
Level 4: ShellRead         --- Shell 只读命令
Level 3: FileWrite         --- 文件写入
Level 2: FileRead          --- 文件读取
Level 1: ReadOnly          --- 只读访问

权限递增方向 ──────────────────────────────────►
┌────────┬────────┬────────┬────────┬────────┬────────┬────────┐
│ReadOnly│File    │File    │Shell   │Shell   │Network │Full    │
│        │Read    │Write   │Read    │Write   │        │Access  │
│        │        │        │        │        │        │        │
│  L1    │  L2    │  L3    │  L4    │  L5    │  L6    │  L7    │
└────────┴────────┴────────┴────────┴────────┴────────┴────────┘
```

**类比**：这就像游戏中的技能树。你不能一开始就点满所有技能，需要一步步解锁。

### 3.2 各级别详细权限

| 级别 | 名称 | 文件读取 | 文件写入 | Shell 只读 | Shell 写入 | 网络 | 典型场景 |
|------|------|---------|---------|-----------|-----------|------|---------|
| L1 | ReadOnly | 允许 | 禁止 | 禁止 | 禁止 | 禁止 | 代码审查、文档阅读 |
| L2 | FileRead | 允许 | 禁止 | 禁止 | 禁止 | 禁止 | 搜索代码、分析依赖 |
| L3 | FileWrite | 允许 | 允许 | 禁止 | 禁止 | 禁止 | 编辑代码、生成文件 |
| L4 | ShellRead | 允许 | 禁止 | 允许 | 禁止 | 禁止 | 运行测试、构建项目 |
| L5 | ShellWrite | 允许 | 允许 | 允许 | 允许 | 禁止 | 完整开发流程 |
| L6 | Network | 允许 | 允许 | 允许 | 允许 | 允许 | 下载依赖、API 调用 |
| L7 | FullAccess | 允许 | 允许 | 允许 | 允许 | 允许 | 完全自动化运维 |

### 3.3 权限提升流程

Agent 默认获得最低权限（L1）。当需要更高权限时，必须经过用户确认：

```
┌─────────────────────────────────────────────────────┐
│                 权限提升流程                          │
│                                                     │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐         │
│  │ Agent   │───→│ 检查    │───→│ 权限    │         │
│  │ 请求    │    │ 当前    │    │ 不足？  │         │
│  └─────────┘    │ 权限    │    └─────────┘         │
│                 └─────────┘         │               │
│                                     ▼               │
│                              ┌─────────────┐        │
│                              │ 发送权限    │        │
│                              │ 请求给用户  │        │
│                              └─────────────┘        │
│                                     │               │
│                         ┌───────────┴───────────┐   │
│                         ▼                       ▼   │
│                  ┌─────────┐             ┌─────────┐│
│                  │ 用户    │             │ 用户    ││
│                  │ 批准    │             │ 拒绝    ││
│                  └─────────┘             └─────────┘│
│                         │                       │   │
│                         ▼                       ▼   │
│                  ┌─────────┐             ┌─────────┐│
│                  │ 提升    │             │ 保持    ││
│                  │ 权限    │             │ 原权限  ││
│                  └─────────┘             └─────────┘│
└─────────────────────────────────────────────────────┘
```

**关键原则**：

- **最小权限**：Agent 默认获得最低权限
- **显式授权**：每级权限提升都需要用户明确确认
- **可审计**：所有权限变更记录可追溯
- **时效性**：权限提升有有效期，过期自动降级

## 4. 规则引擎 - DSL 语法设计

### 4.1 为什么需要规则引擎？

7 级权限模型解决了"Agent 能做什么"的问题，但还有一个问题：**Agent 对哪些资源能做什么？**

**类比**：

- 7 级权限 = 公司的职级（决定了你能进入哪些区域）
- 规则引擎 = 部门的具体规章制度（决定了你在某个区域能做什么）

比如，一个 L3（FileWrite）级别的 Agent：

- 可以写 `src/` 目录下的文件
- 不能写 `target/` 目录（编译输出）
- 不能写 `.env` 文件（敏感配置）

这就是规则引擎的作用。

### 4.2 规则文件结构

规则文件存放在 `.codey/rules/` 目录下，使用 `.rules` 扩展名：

```
.codey/
  rules/
    default.rules       # 默认规则
    src.rules           # 源代码目录规则
    config.rules        # 配置文件规则
    test.rules          # 测试目录规则
    deploy.rules        # 部署相关规则
```

### 4.3 DSL 语法示例

规则引擎使用自定义的 DSL（Domain Specific Language）语法：

```
# .codey/rules/default.rules
@version "1.0"
@description "CodeY 默认权限规则"

# -- 全局默认 -------------------------------------------

rule default_readonly {
  when path matches "**"
  then set level ReadOnly
  priority 0
}

# -- 源代码规则 -----------------------------------------

rule source_read {
  when path matches "src/**" and action is "read"
  then allow
  priority 10
}

rule source_write {
  when path matches "src/**" and action is "write"
  then allow with level ReadWrite
  priority 10
}

# -- 敏感文件保护 ---------------------------------------

rule protect_env_files {
  when path matches ".env*" and action is "write"
  then deny with reason "禁止修改环境变量文件，请手动配置"
  priority 200
}

rule protect_secrets {
  when path matches "**/*secret*" or path matches "**/*key*"
  then deny with reason "禁止访问密钥相关文件"
  priority 200
}

# -- 审计日志 -------------------------------------------

rule audit_all_writes {
  when action is "write" or action is "delete"
  then audit
  priority 999
}
```

### 4.4 规则匹配流程

当 Agent 发起操作时，规则引擎按以下流程处理：

```
┌─────────────────────────────────────────────────────┐
│                 规则匹配流程                          │
│                                                     │
│  ┌─────────────┐                                    │
│  │ 收到操作请求 │                                    │
│  └──────┬──────┘                                    │
│         │                                           │
│         ▼                                           │
│  ┌─────────────────────────────────────────────┐    │
│  │  遍历所有规则文件，收集匹配的规则            │    │
│  │  (按 priority 数值从高到低排序)              │    │
│  └──────┬──────────────────────────────────────┘    │
│         │                                           │
│         ▼                                           │
│  ┌──────────────────────┐    是    ┌───────────┐    │
│  │ 是否存在 deny 规则？  │────────>│ 拒绝操作   │    │
│  └──────┬───────────────┘         └───────────┘    │
│         │ 否                                        │
│         ▼                                           │
│  ┌──────────────────────┐    是    ┌───────────┐    │
│  │ 是否存在 allow 规则？ │────────>│ 允许操作   │    │
│  └──────┬───────────────┘         └───────────┘    │
│         │ 否                                        │
│         ▼                                           │
│  ┌──────────────────────┐                           │
│  │ 使用 default 规则    │                           │
│  │ 或拒绝（安全回退）    │                           │
│  └──────────────────────┘                           │
└─────────────────────────────────────────────────────┘
```

**冲突解决原则**：

1. **deny 优先于 allow**（安全第一）
2. **priority 数值越大，优先级越高**
3. **相同 priority 时，deny > allow**
4. **无匹配规则时，默认拒绝（deny by default）**

### 4.5 条件表达式

规则引擎支持丰富的条件表达式：

| 条件类型 | 示例 | 说明 |
|---------|------|------|
| 路径匹配 | `path matches "src/**/*.rs"` | 支持 Glob 模式 |
| 操作类型 | `action is "read"` | read/write/execute/delete |
| 复合条件 | `path matches "src/**" and action is "read"` | and/or/not 组合 |
| 上下文条件 | `agent.name is "codey-agent"` | Agent 信息 |
| 时间条件 | `time.hour >= 9 and time.hour <= 18` | 时间限制 |

## 5. 沙箱系统 - OS 级隔离

### 5.1 为什么需要沙箱？

权限模型和规则引擎都是"软件层面"的限制。但如果 Agent 被攻破了呢？或者 Agent 执行了一个危险的系统调用呢？

**类比**：

- 权限模型 = 公司的规章制度（软约束）
- 沙箱系统 = 公司的物理门禁（硬约束）

即使有人违反了规章制度，物理门禁也会阻止他们进入禁区。

### 5.2 平台架构差异

不同平台有不同的沙箱机制：

```
┌─────────────────────────────────────────────────────┐
│                 Desktop 沙箱 (OS-level)              │
│                                                     │
│  ┌──────────────┐    ┌──────────────┐    ┌────────┐│
│  │   Agent      │    │   Host       │    │   OS   ││
│  │   Process    │◄──►│   Process    │◄──►│ Kernel ││
│  └──────────────┘    └──────────────┘    └────────┘│
│                                                     │
│  沙箱机制：macOS Seatbelt / Linux seccomp           │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│                 Web 沙箱 (Server-side)               │
│                                                     │
│  ┌──────────────┐    ┌──────────────┐    ┌────────┐│
│  │   Browser    │    │   Server     │    │Docker  ││
│  │   Client     │◄──►│   Gateway    │◄──►│Container│
│  └──────────────┘    └──────────────┘    └────────┘│
│                                                     │
│  沙箱机制：Docker container + namespace + cgroup    │
└─────────────────────────────────────────────────────┘
```

### 5.3 沙箱配置对比

| 特性 | Desktop (macOS) | Desktop (Linux) | Web (Docker) |
|------|----------------|-----------------|--------------|
| 隔离级别 | 进程级 | 进程级 | 容器级 |
| 文件系统 | Seatbelt profile | seccomp + mount ns | bind mount + tmpfs |
| 网络 | 应用层过滤 | iptables/nftables | Docker network |
| 资源限制 | launchd limits | cgroups v2 | Docker limits |
| 性能开销 | 极低 | 低 | 中等 |

### 5.4 容器隔离层级

Web 环境下，Agent 运行在 Docker 容器中，通过 Linux Namespace 和 Cgroup 实现隔离：

```
容器隔离层级：

┌─────────────────────────────────────────────┐
│  Host OS                                    │
│  ┌─────────────────────────────────────────┐│
│  │  Docker Engine                          ││
│  │  ┌─────────────────────────────────────┐││
│  │  │  Container (sandbox)                │││
│  │  │                                     │││
│  │  │  Namespaces:                        │││
│  │  │    PID    - 进程隔离                │││
│  │  │    NET    - 网络隔离                │││
│  │  │    MNT    - 文件系统隔离            │││
│  │  │    UTS    - 主机名隔离              │││
│  │  │    IPC    - 进程间通信隔离          │││
│  │  │    USER   - 用户隔离                │││
│  │  │                                     │││
│  │  │  Cgroups:                           │││
│  │  │    CPU    - 限制 1 core             │││
│  │  │    Memory - 限制 512MB              │││
│  │  │    PID    - 最多 100 进程           │││
│  │  │    IO     - 限制磁盘 I/O           │││
│  │  │                                     │││
│  │  │  Filesystem:                        │││
│  │  │    /workspace  - 项目目录（读写）    │││
│  │  │    /tmp        - tmpfs（100MB）      │││
│  │  │    /           - 只读 rootfs         │││
│  │  │                                     │││
│  │  └─────────────────────────────────────┘││
│  └─────────────────────────────────────────┘│
└─────────────────────────────────────────────┘
```

**关键配置**：

| 配置项 | 值 | 作用 |
|--------|-----|------|
| `read_only` | true | 根文件系统只读，防止篡改 |
| `cap_drop` | ALL | 移除所有 Linux capabilities |
| `mem_limit` | 512m | 内存上限，防止内存泄漏 |
| `pids_limit` | 100 | 进程数上限，防止 fork bomb |
| `internal` | true | 网络隔离，禁止外部访问 |

## 6. 实现细节 - 少代码，多流程图

### 6.1 完整权限检查链路

当 Agent 发起操作时，系统按以下流程进行权限检查：

```
Agent 发起操作
     │
     ▼
┌─────────────────┐
│ 1. 解析操作类型  │  识别 action (read/write/execute/...)
│    和目标路径    │  识别 target path
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 2. 查询权限级别  │  检查当前 Agent 的 permission level
└────────┬────────┘
         │
         ▼
┌─────────────────┐    权限不足    ┌─────────────────┐
│ 3. 基础权限检查  │──────────────►│ 返回 Permission  │
│    (Level >=     │               │ Denied (-32001)  │
│     Required)    │               │ 并建议所需级别    │
└────────┬────────┘               └─────────────────┘
         │ 权限足够
         ▼
┌─────────────────┐    匹配 deny   ┌─────────────────┐
│ 4. 规则引擎检查  │──────────────►│ 返回规则拒绝原因  │
│    (.rules)      │               │ (附带规则名)      │
└────────┬────────┘               └─────────────────┘
         │ 匹配 allow / 无匹配
         ▼
┌─────────────────┐    需要确认    ┌─────────────────┐
│ 5. 确认检查      │──────────────►│ 发送 permission  │
│    (confirmation)│               │ .request 给用户   │
└────────┬────────┘               └────────┬────────┘
         │ 已确认/无需确认                   │
         │                                 ▼
         │                          ┌─────────────────┐
         │                          │ 用户批准/拒绝    │
         │                          └────────┬────────┘
         │                          批准 ────┘───► 拒绝
         │                            │           │
         ▼                            │           ▼
┌─────────────────┐                   │    ┌──────────────┐
│ 6. 沙箱执行      │◄──────────────────┘    │ 返回拒绝      │
└────────┬────────┘                        └──────────────┘
         │
         ▼
┌─────────────────┐
│ 7. 审计日志记录  │  记录操作、结果、时间戳
└────────┬────────┘
         │
         ▼
    返回操作结果
```

### 6.2 权限检查的核心逻辑

权限检查的核心逻辑可以用一句话概括：**三重验证，deny 优先**。

```
┌─────────────────────────────────────────────────────┐
│                 权限检查核心逻辑                      │
│                                                     │
│  第一重：基础级别检查                                │
│  ┌─────────────────────────────────────────────┐    │
│  │ Agent 的权限级别 >= 操作所需级别？            │    │
│  │ 否 -> 拒绝，返回错误码 -32001                 │    │
│  └─────────────────────────────────────────────┘    │
│                      │ 通过                         │
│                      ▼                              │
│  第二重：规则引擎检查                                │
│  ┌─────────────────────────────────────────────┐    │
│  │ 加载 .rules 文件，按 priority 排序           │    │
│  │ 匹配 deny 规则？-> 拒绝，返回规则名            │    │
│  │ 匹配 allow 规则？-> 继续                      │    │
│  │ 无匹配？-> 使用 default 或拒绝                 │    │
│  └─────────────────────────────────────────────┘    │
│                      │ 通过                         │
│                      ▼                              │
│  第三重：沙箱检查                                    │
│  ┌─────────────────────────────────────────────┐    │
│  │ 操作是否被 OS 级沙箱策略允许？                │    │
│  │ 否 -> 拒绝，返回沙箱拒绝原因                   │    │
│  └─────────────────────────────────────────────┘    │
│                      │ 通过                         │
│                      ▼                              │
│  ┌─────────────────────────────────────────────┐    │
│  │ 记录审计日志，返回允许                        │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 6.3 配置层级

权限系统支持多层配置，越具体的配置优先级越高：

```
┌─────────────────────────────────────────────────────┐
│                 配置层级                              │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │
│  │  用户级配置 (~/.codey/config.toml)          │    │
│  │  全局默认值、信任策略、通知方式               │    │
│  └─────────────────────────────────────────────┘    │
│                      │ 覆盖                        │
│                      ▼                              │
│  ┌─────────────────────────────────────────────┐    │
│  │  项目级配置 (.codey/config.toml)            │    │
│  │  项目特定的权限、规则、沙箱配置               │    │
│  └─────────────────────────────────────────────┘    │
│                      │ 覆盖                        │
│                      ▼                              │
│  ┌─────────────────────────────────────────────┐    │
│  │  运行时动态配置                              │    │
│  │  临时权限提升、权限撤销                       │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 6.4 权限 Scope 映射

每个权限级别由多个 scope 组合而成，scope 到级别的映射关系：

```
L1 ReadOnly:         file.read, file.list
L2 FileRead:         L1 + (额外的读取能力)
L3 FileWrite:        L1 + file.write, file.delete
L4 ShellRead:        L1 + shell.exec(受限)
L5 ShellWrite:       L3 + L4 (合并)
L6 Network:          L5 + network.http, network.socket
L7 FullAccess:       L6 + hardware.gpu, hardware.storage
```

## 8. 总结 - 学到了什么

### 8.1 设计原则回顾

通过设计 CodeY 的权限系统，我们学到了几个重要的设计原则：

```
┌─────────────────────────────────────────────────────┐
│                 设计原则                              │
│                                                     │
│  1. 最小权限原则                                     │
│     Agent 默认获得最低权限，按需逐级提升             │
│                                                     │
│  2. 纵深防御                                         │
│     多层权限检查，任何一层失败都阻止操作             │
│                                                     │
│  3. deny 优先                                        │
│     安全第一，拒绝优先于允许                         │
│                                                     │
│  4. 可审计                                           │
│     所有权限变更和操作都记录在案                     │
│                                                     │
│  5. 用户控制                                         │
│     敏感操作必须经过用户确认                         │
└─────────────────────────────────────────────────────┘
```

### 8.2 架构优势

| 优势 | 说明 |
|------|------|
| 灵活性 | 7 级权限 + 规则引擎，可以精细控制 |
| 安全性 | 三重验证 + deny 优先 + OS 级沙箱 |
| 可扩展性 | 规则文件可以按模块组织，易于维护 |
| 可审计性 | 完整的审计日志，便于追溯问题 |
| 跨平台 | Desktop 和 Web 使用不同的沙箱策略 |

### 8.3 未来展望

权限系统还有很多可以改进的地方：

1. **机器学习辅助**：通过分析 Agent 的行为模式，自动调整权限策略
2. **动态规则**：根据项目状态（如是否在 CI/CD 环境）动态调整规则
3. **权限继承**：支持团队级别的权限模板，减少重复配置
4. **可视化管理**：提供 Web UI 来管理权限规则和审计日志
