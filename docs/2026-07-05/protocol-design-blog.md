# CodeY 协议设计：从混乱到统一的思考之路

> 当你面对一个需要同时支持桌面应用、Web 前端、CLI 工具的系统时，如何设计一套能够统一所有通信的协议？这篇文章记录了 CodeY 在协议设计过程中的思考和决策。

## 1. 引言

### 为什么需要统一协议？

想象一下这个场景：你在开发一个 AI 编程助手，它需要：
- 在桌面应用中与后端通信
- 在浏览器中连接远程服务
- 在命令行中执行文件操作
- 处理流式响应、权限控制、错误恢复...

如果没有统一的协议，你可能会写出这样的代码：

```javascript
// 桌面端用 Tauri IPC
const result = await invoke('read_file', { path: '/foo' });

// Web 端用 WebSocket
ws.send(JSON.stringify({ type: 'read_file', path: '/foo' }));

// CLI 用 HTTP
const result = await fetch('/api/read_file', { body: JSON.stringify({ path: '/foo' }) });
```

三套代码，三种错误处理，三种超时逻辑。维护成本可想而知。

### CodeY 的协议设计目标

```
┌─────────────────────────────────────────────────────────────────┐
│                    CodeY 协议设计目标                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   1. 统一性：一套协议，多种传输层                                 │

│   3. 类型安全：TypeScript 类型贯穿始终                           │
│   4. 流式支持：原生支持 SSE 和流式响应                           │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 2. 协议选型思考

### JSON-RPC 2.0 vs REST vs GraphQL

在开始设计之前，我们评估了三种主流方案：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        协议对比矩阵                                  │
├─────────────┬─────────────────┬─────────────────┬─────────────────┤
│    特性      │   JSON-RPC 2.0  │      REST       │    GraphQL      │
├─────────────┼─────────────────┼─────────────────┼─────────────────┤
│ 双向通信     │      ✓         │       ✗         │       ✗         │
│ 批量请求     │      ✓         │       ✗         │       ✓         │
│ 类型安全     │      ✓         │       需额外配置  │       ✓         │
│ 流式支持     │      ✓         │       需额外实现  │       需订阅     │
│ 学习曲线     │      低         │       低         │       中         │
│ 适用场景     │ 内部服务通信     │ 公开 API        │ 复杂查询         │
└─────────────┴─────────────────┴─────────────────┴─────────────────┘
```

### 为什么选择 JSON-RPC 2.0？

**1. 双向通信支持**

JSON-RPC 天然支持 Request-Response 模式，同时也支持 Notification（无需响应的消息）。这对于 AI 场景至关重要——用户发送消息（Request），AI 流式返回（Notification 批量）。

```
用户                    服务端
  │                       │
  │   Request (id: 1)     │
  │ ────────────────────> │
  │                       │
  │   Notification        │
  │ <──────────────────── │  (流式 chunk 1)
  │                       │
  │   Notification        │
  │ <──────────────────── │  (流式 chunk 2)
  │                       │
  │   Response (id: 1)    │
  │ <──────────────────── │  (最终结果)
  │                       │
```

**2. 简单轻量**

JSON-RPC 2.0 规范只有几页，核心概念就三个：Request、Response、Notification。没有 REST 的资源概念，没有 GraphQL 的 Schema 定义，上手极快。

**3. 广泛支持**

几乎所有语言都有成熟的 JSON-RPC 库。TypeScript 有 `vscode-jsonrpc`，Rust 有 `jsonrpsee`，Go 有标准库支持。

## 3. 消息格式设计

### 核心消息类型

我们定义了三种消息类型，每种都有明确的用途：

```
┌─────────────────────────────────────────────────────────────────┐
│                       消息格式设计                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Request (请求)                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                       │   │
│  │   "jsonrpc": "2.0",        // 协议版本                   │   │
│  │   "id": 1,                 // 请求 ID (用于匹配响应)      │   │
│  │   "method": "agent.chat",  // 方法名                     │   │
│  │   "params": { ... }        // 参数                       │   │
│  │ }                                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Response (响应)                                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                       │   │
│  │   "jsonrpc": "2.0",                                    │   │
│  │   "id": 1,                 // 对应的请求 ID              │   │
│  │   "result": { ... }        // 成功结果 (或 error)        │   │
│  │ }                                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  Notification (通知，无需响应)                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ {                                                       │   │
│  │   "jsonrpc": "2.0",                                    │   │
│  │   "method": "progress",    // 没有 id 字段               │   │
│  │   "params": { ... }                                    │   │
│  │ }                                                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 设计决策：为什么这样划分？

- **Request 有 id**：用于异步匹配响应，支持并发请求
- **Response 有 id**：明确告诉客户端这是哪个请求的响应
- **Notification 无 id**：明确表示"我不需要你回复"，适合流式推送、进度更新等场景

## 4. 方法分类设计

### 四大方法域

我们将所有 RPC 方法按功能划分为四个域：

```
┌─────────────────────────────────────────────────────────────────┐
│                       方法分类架构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐       │
│   │    Agent      │   │     File     │   │    Shell     │       │
│   │   会话管理     │   │   文件操作    │   │   命令执行    │       │
│   ├──────────────┤   ├──────────────┤   ├──────────────┤       │
│   │ chat         │   │ read         │   │ execute      │       │
│   │ cancel       │   │ write        │   │ cancel       │       │
│   │ history      │   │ list         │   │ status       │       │
│   │ models       │   │ watch        │   │ stream       │       │
│   └──────────────┘   └──────────────┘   └──────────────┘       │
│                                                                 │
│                     ┌──────────────┐                            │
│                     │  Permission  │                            │
│                     │   权限控制    │                            │
│                     ├──────────────┤                            │
│                     │ check        │                            │
│                     │ request      │                            │
│                     │ revoke       │                            │
│                     └──────────────┘                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 为什么这样分类？

**1. Agent 方法——会话管理**

AI 对话是核心功能，但"对话"不只是发送消息：
- 需要管理多个会话（`create_session`, `list_sessions`）
- 需要取消正在生成的响应（`cancel`）
- 需要切换模型（`set_model`）
- 需要查看历史（`history`）

**2. File 方法——文件操作**

文件操作看似简单，但要考虑：
- 大文件的分块读取（`read` with `offset`/`limit`）
- 文件变更监听（`watch`）
- 原子写入（`write` with `createDirs`）
- 路径验证（防止目录遍历攻击）

**3. Shell 方法——命令执行**

命令执行是最危险的操作，需要：
- 沙箱执行（限制可执行命令）
- 超时控制（防止挂起）
- 流式输出（实时显示）
- 取消支持（用户中断）

**4. Permission 方法——权限控制**

安全性是重中之重：
- 细粒度权限检查（每个操作前）
- 权限请求流程（用户确认）
- 权限撤销（会话结束时清理）

## 5. 错误处理策略

### 标准错误码 vs 自定义错误码

JSON-RPC 2.0 定义了标准错误码，但我们还需要自定义错误码来覆盖特定场景：

```
┌─────────────────────────────────────────────────────────────────┐
│                       错误码设计                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  标准错误码 (JSON-RPC 2.0 规范)                                  │
│  ┌─────────┬─────────────────────────────────────────────────┐ │
│  │ -32700  │ Parse error (JSON 解析失败)                      │ │
│  │ -32600  │ Invalid Request (无效请求)                       │ │
│  │ -32601  │ Method not found (方法不存在)                    │ │
│  │ -32602  │ Invalid params (参数错误)                        │ │
│  │ -32603  │ Internal error (内部错误)                        │ │
│  └─────────┴─────────────────────────────────────────────────┘ │
│                                                                 │
│  自定义错误码 (CodeY 扩展)                                       │
│  ┌─────────┬─────────────────────────────────────────────────┐ │
│  │ -32000  │ Permission denied (权限被拒绝)                   │ │
│  │ -32001  │ Resource not found (资源不存在)                  │ │
│  │ -32002  │ Timeout (操作超时)                               │ │
│  │ -32003  │ Rate limited (请求过于频繁)                      │ │
│  │ -32004  │ Session expired (会话已过期)                     │ │
│  │ -32005  │ Model error (模型调用失败)                       │ │
│  └─────────┴─────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 错误恢复机制设计

```
客户端                        服务端
  │                             │
  │  Request (method: X)        │
  │ ──────────────────────────> │
  │                             │
  │  Error (code: -32002)       │
  │ <────────────────────────── │  超时
  │                             │
  │  自动重试 (指数退避)          │
  │  Request (method: X)        │
  │ ──────────────────────────> │
  │                             │
  │  Result                     │
  │ <────────────────────────── │  成功
  │                             │
```

**重试策略：**
- 网络错误、超时：自动重试（指数退避）
- 权限错误：提示用户授权
- 业务逻辑错误：不重试，返回错误

### 用户友好的错误消息

错误不仅要机器可读，更要人可读：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32000,
    "message": "Permission denied",
    "data": {
      "permission": "file:write",
      "path": "/etc/passwd",
      "userMessage": "没有写入 /etc/passwd 的权限，请在设置中授权文件写入。",
      "suggestion": "点击「授权」按钮，或在设置中添加该路径到白名单。"
    }
  }
}
```

## 6. 流式传输设计

### 为什么需要流式？

AI 生成文本是逐 token 进行的，如果等到全部生成完再返回，用户需要等待数十秒。流式传输让用户看到"打字机效果"，体验大幅提升。

```
┌─────────────────────────────────────────────────────────────────┐
│                    流式传输 vs 批量传输                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  批量传输：用户等待 10 秒...                                      │
│  ┌────────────────────────────────────────────────────────────┐│
│  │ [长时间等待] ─────────────────────────────────> 完整结果    ││
│  └────────────────────────────────────────────────────────────┘│
│                                                                 │
│  流式传输：用户立即看到反馈                                       │
│  ┌────────────────────────────────────────────────────────────┐│
│  │ [0.1s] ──> "你"  [0.2s] ──> "好"  [0.3s] ──> "！"  ...   ││
│  └────────────────────────────────────────────────────────────┘│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### SSE vs WebSocket 选型

```
┌─────────────────────────────────────────────────────────────────┐
│                    传输协议对比                                   │
├─────────────┬─────────────────┬─────────────────┬─────────────┤
│    特性      │      SSE        │   WebSocket     │   选型      │
├─────────────┼─────────────────┼─────────────────┼─────────────┤
│ 方向         │   单向 (服务端→客户端) │     双向         │  SSE       │
│ 协议         │      HTTP       │    WebSocket     │  SSE       │
│ 自动重连     │      ✓         │       ✗         │  SSE       │
│ 穿透代理     │      ✓         │       需配置      │  SSE       │
│ 浏览器支持   │      ✓         │       ✓         │  平手       │
│ 复杂度       │      低         │       高         │  SSE       │
└─────────────┴─────────────────┴─────────────────┴─────────────┘

结论：AI 场景主要是服务端推送给客户端，SSE 完全够用且更简单。
```

### 流式消息格式

```json
// 流式 chunk (Notification)
{"jsonrpc":"2.0","method":"stream.chunk","params":{"sessionId":"abc","delta":"你好"}}

// 流式完成 (Notification)
{"jsonrpc":"2.0","method":"stream.done","params":{"sessionId":"abc","fullText":"你好！有什么可以帮你的？"}}

// 流式错误 (Notification)
{"jsonrpc":"2.0","method":"stream.error","params":{"sessionId":"abc","error":{"code":-32005,"message":"模型调用失败"}}}
```

## 7. 传输层抽象

### 统一协议，多种传输

这是 CodeY 协议设计的核心理念：**协议层与传输层完全解耦**。

```
┌─────────────────────────────────────────────────────────────────┐
│                    传输层抽象架构                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                    Protocol Layer                        │  │
│   │              (JSON-RPC 2.0 消息处理)                     │  │
│   │                                                         │  │
│   │    Request ──> Method Dispatch ──> Response              │  │
│   └─────────────────────────────────────────────────────────┘  │
│                              │                                  │
│                              │ 统一接口                          │
│                              ▼                                  │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                   Transport Layer                        │  │
│   ├─────────────┬─────────────┬─────────────┬─────────────┐  │  │
│   │   Tauri     │  WebSocket  │ HTTP POST   │    SSE      │  │  │
│   │    IPC      │             │             │             │  │  │
│   ├─────────────┼─────────────┼─────────────┼─────────────┤  │  │
│   │   桌面应用   │   Web 前端   │  简单场景    │  服务端推送  │  │  │
│   └─────────────┴─────────────┴─────────────┴─────────────┘  │  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 各传输层特点

**1. Tauri IPC (Desktop)**
- 零网络开销，直接内存通信
- 类型安全，编译时检查
- 适合桌面应用场景

**2. WebSocket (Web)**
- 全双工通信
- 支持双向消息
- 适合需要实时交互的 Web 场景

**3. HTTP POST (简单场景)**
- 最简单的实现
- 无状态，易于负载均衡
- 适合不需要流式的简单调用

**4. SSE (服务端推送)**
- 基于 HTTP，穿透性好
- 自动重连
- 适合 AI 流式响应

### 如何实现传输层切换？

```typescript
// 使用示例：开发者无需关心底层传输
const client = new CodeYClient({
  transport: 'tauri',  // 或 'websocket', 'http', 'sse'
  // 其他配置...
});

// 统一的 API 调用
const result = await client.call('file.read', { path: '/foo' });
```

底层实现会根据配置自动选择传输层，但对外暴露的 API 完全一致。

## 8. 安全考虑

### 权限检查集成

```
┌─────────────────────────────────────────────────────────────────┐
│                    权限检查流程                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   Client Request                                                │
│        │                                                        │
│        ▼                                                        │
│   ┌─────────────┐                                               │
│   │ 方法是否存在 │ ─── No ──> Error: Method not found            │
│   └─────────────┘                                               │
│        │ Yes                                                    │
│        ▼                                                        │
│   ┌─────────────┐                                               │
│   │ 参数是否合法 │ ─── No ──> Error: Invalid params              │
│   └─────────────┘                                               │
│        │ Yes                                                    │
│        ▼                                                        │
│   ┌─────────────┐                                               │
│   │ 权限是否足够 │ ─── No ──> Error: Permission denied           │
│   └─────────────┘                                               │
│        │ Yes                                                    │
│        ▼                                                        │
│   执行方法并返回结果                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 沙箱执行

对于 Shell 命令执行，必须在沙箱中运行：

- **命令白名单**：只允许执行预定义的安全命令
- **路径限制**：禁止访问敏感目录（如 `/etc`, `/usr`）
- **超时控制**：防止命令挂起
- **资源限制**：限制 CPU、内存使用

### 输入验证

所有来自客户端的输入都必须验证：

- **JSON Schema 验证**：确保参数类型正确
- **路径规范化**：防止目录遍历（`../../etc/passwd`）
- **长度限制**：防止内存溢出
- **特殊字符转义**：防止注入攻击

### MCP 扩展支持

MCP (Model Context Protocol) 是 Anthropic 提出的开放协议，用于 AI 工具集成。CodeY 计划在 Phase 2 中支持 MCP，让 CodeY 能够：
- 作为 MCP Server 被其他 AI 工具调用
- 作为 MCP Client 调用其他工具服务
- 实现工具生态的互联互通

### 性能优化

- **连接复用**：减少建立连接的开销
- **消息批量化**：合并多个小请求
- **压缩传输**：对大消息启用 gzip 压缩
- **懒加载**：按需加载不常用的方法定义

## 总结

CodeY 的协议设计遵循了以下原则：

1. **简单优先**：选择 JSON-RPC 2.0 是因为它的简单性，而不是功能丰富性
2. **关注点分离**：协议层与传输层解耦，让同一套代码支持多种场景
3. **安全第一**：每个操作都经过权限检查，危险操作在沙箱中执行
4. **用户体验**：流式传输、友好错误消息、自动重试——让用户感受不到协议的存在

好的协议设计应该是透明的——用户不需要知道底层用的是 JSON-RPC 还是 GraphQL，他们只需要知道"它能工作，而且工作得很好"。
