# codey-brainstorming 完整文档

定义 CodeY 项目的头脑风暴流程，用于需求收集、研究调研和 Spec 生成。

---

## 核心原则（Core Principles）

### 1. Hybrid Brainstorming Mode（混合头脑风暴模式）

结合问答式和多方案展示两种方式：

- **Q&A 式**：用户描述需求，Agent 通过提问澄清模糊点
- **多方案展示**：Agent 提供多个可行方案，用户选择最佳方案
- **知识互补**：用户的知识盲区由 Agent 的专业知识填补
- **协作决策**：最终决策权始终在用户手中

**交互原则**

| 原则 | 说明 |
|------|------|
| 先理解后实现 | 充分理解需求后再动手 |
| 先提问后假设 | 有疑问就提问，不做假设 |
| 先 Spec 后代码 | Spec 确认后才开始写代码 |
| 多方案对比 | 每个决策点提供 2-3 个方案 |
| 用户决策 | Agent 提供分析，用户做出选择 |

---

### 2. Research Integration（研究整合）

在头脑风暴过程中，必须进行充分的研究调研。

**研究工具优先级**

```
1. GitHub 代码搜索   → 查找类似实现、参考项目、代码模式
2. Context7 文档查询  → 查询库/框架的官方文档和示例
3. Exa 网络搜索      → 更广泛的网络研究、最佳实践、技术文章
```

**研究内容**

| 研究类型 | 目的 | 工具 |
|----------|------|------|
| 类似实现 | 找到可参考的开源项目 | GitHub Search |
| 最佳实践 | 了解行业标准做法 | Exa Search |
| 库文档 | 确认 API 行为和版本差异 | Context7 |
| 技术方案 | 收集可行的技术选型 | 综合使用 |
| 安全考量 | 识别潜在安全风险 | Exa + GitHub |

**研究时机**

- **Phase 2（研究调研）**：集中进行研究
- **Phase 3（方案生成）**：针对特定方案补充研究
- **Phase 4（细节确认）**：确认技术细节时按需研究

---

### 3. Spec Generation（Spec 生成）

头脑风暴完成后，必须生成 Spec 文档。

**目录结构**

```
docs/
└── specs/
    └── YYYY-MM-DD-<feature>/
        ├── design.md          # 设计文档：架构决策、数据模型、技术选型
        ├── api.md             # API 规范：接口定义、请求/响应格式、错误码
        └── test.md            # 测试规范：测试策略、用例清单、覆盖率目标
```

**命名规范**

- 目录名：`YYYY-MM-DD-<feature>`，如 `2026-07-05-user-auth`
- 文件名：固定为 `design.md`、`api.md`、`test.md`
- 特性名使用小写连字符：`user-auth`、`file-upload`、`agent-protocol`

## 问题类型（Question Types）

### 1. Clarifying Questions（澄清问题）

用于理解需求的本质和边界：

- "这个功能的主要目的是什么？"
- "用户使用这个功能的典型场景是什么？"
- "有哪些约束条件？"
- "优先级如何？MVP 包含哪些内容？"
- "与现有功能的关系是什么？"
- "有参考实现或对标产品吗？"

详见：[questions/clarifying.md](./questions/clarifying.md)

### 2. Technical Questions（技术问题）

用于明确技术要求和限制：

- "你倾向于哪种技术方案？"
- "有哪些性能要求？（响应时间、并发量）"
- "有哪些安全要求？（认证、加密、权限）"
- "需要支持哪些平台/浏览器？"
- "有哪些兼容性要求？"
- "数据量预估是多少？"

详见：[questions/technical.md](./questions/technical.md)

### 3. Trade-off Questions（权衡问题）

当存在多种方案时，帮助用户做出选择：

- "方案 A 优点是...缺点是..."
- "方案 B 优点是...缺点是..."
- "你更看重哪个方面？（性能 vs 开发速度）"
- "可以接受的复杂度上限是什么？"

详见：[questions/tradeoff.md](./questions/tradeoff.md)

## 完整示例

### 场景：实现用户认证功能

**Phase 1: 需求收集**

```
用户：我想给 CodeY 添加用户认证功能

Agent：好的，我需要了解一些细节：

1. 认证方式：你倾向于哪种认证方式？
   - 方案 A: JWT Token（无状态，适合分布式）
   - 方案 B: Session + Cookie（有状态，传统方式）
   - 方案 C: OAuth 2.0（第三方登录）

2. 用户存储：用户数据存储在哪里？
   - 本地数据库
   - 第三方服务（如 Firebase Auth）

3. 安全要求：
   - 密码加密方式？（bcrypt, argon2）
   - 需要双因素认证吗？
   - 需要记住登录状态吗？

4. 功能范围：
   - MVP 包含哪些功能？（注册、登录、登出、密码重置）
   - 需要用户资料管理吗？
   - 需要权限控制吗？

用户：
- 选择 JWT Token
- 本地数据库存储
- 使用 bcrypt 加密，MVP 不需要 2FA
- MVP 包含注册、登录、登出，不需要密码重置
```

**Phase 2: 研究调研**

```
Agent 进行研究：

GitHub 搜索：
- next-auth: 15k+ stars，Next.js 认证方案
- lucia: 2k+ stars，轻量级认证库
- better-auth: 1k+ stars，类型安全的认证库

最佳实践：
- JWT 最佳实践：短过期时间 + Refresh Token
- 密码存储：bcrypt 成本因子 >= 12
- API 安全：Rate limiting, CORS, CSP

库文档（Context7）：
- bcryptjs: hash, compare 方法
- jsonwebtoken: sign, verify 方法
- prisma: User 模型定义
```

**Phase 3: 方案生成**

```
Agent 展示方案：

## 方案对比

| 维度 | 方案 A: JWT + Refresh Token | 方案 B: JWT 无 Refresh | 方案 C: Session |
|------|---------------------------|----------------------|-----------------|
| 技术栈 | JWT + Redis | JWT | Session + Cookie |
| 开发复杂度 | 中 | 低 | 低 |
| 安全性 | 高 | 中 | 高 |
| 可扩展性 | 好 | 好 | 差 |
| 维护成本 | 中 | 低 | 低 |

推荐方案 A，理由：
1. 安全性最高（短 Token + Refresh Token）
2. 适合分布式架构
3. 用户体验好（无频繁登录）

用户：选择方案 A
```

**Phase 4: 细节确认**

```
Agent 确认细节：

技术细节：
- [x] 数据模型：User 表（id, email, password_hash, created_at, updated_at）
- [x] Token 策略：Access Token 15分钟，Refresh Token 7天
- [x] 错误处理：统一错误码，不泄露敏感信息
- [x] 性能：bcrypt 并发限制，Redis 缓存 Token

接口设计：
- [x] POST /api/auth/register
- [x] POST /api/auth/login
- [x] POST /api/auth/logout
- [x] POST /api/auth/refresh

测试策略：
- [x] 单元测试：密码加密、Token 生成/验证
- [x] 集成测试：认证流程、错误处理
- [x] E2E 测试：注册 -> 登录 -> 访问受保护资源

用户：确认无误
```

**Phase 5: Spec 生成**

```
Agent 生成 Spec 文档：

docs/specs/2026-07-05-user-auth/
├── design.md     # 架构设计、数据模型、技术选型
├── api.md        # API 端点、请求/响应格式、错误码
└── test.md       # 测试策略、用例清单、覆盖率目标

用户审批后，标记 Spec 为 approved
```

## 最佳实践

1. **充分提问**：不要假设，有疑问就问
2. **多方案展示**：每个决策点提供 2-3 个方案
3. **研究先行**：先研究再方案，不要凭空设计
4. **用户决策**：Agent 提供分析，用户做出选择
5. **Spec 确认**：Spec 必须用户确认后才进入开发
6. **中文交流**：技术术语使用英文，说明使用中文
7. **记录研究**：研究结果记录下来，方便后续参考
8. **渐进细化**：从粗到细，逐步确认细节
