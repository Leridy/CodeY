# Trade-off Questions（权衡问题）

## 用途

当存在多种方案时，帮助用户做出选择。在方案生成阶段（Phase 3）使用，帮助用户理解各方案的优缺点。

## 问题模板

### 方案对比

```
- "方案 A 优点是...缺点是..."
- "方案 B 优点是...缺点是..."
- "方案 C 优点是...缺点是..."
- "你更看重哪个方面？"
```

### 权衡维度

```
- "性能 vs 开发速度：你更看重哪个？"
- "安全性 vs 用户体验：你更看重哪个？"
- "可扩展性 vs 简单性：你更看重哪个？"
- "功能完整性 vs MVP 速度：你更看重哪个？"
```

### 复杂度评估

```
- "可以接受的复杂度上限是什么？"
- "团队的技术能力如何？"
- "学习曲线可以接受吗？"
- "维护成本可以接受吗？"
```

### 风险评估

```
- "可以接受的技术风险是什么？"
- "如果方案失败，有什么备选方案？"
- "时间线紧张吗？可以接受延期吗？"
- "预算紧张吗？可以接受超支吗？"
```

## 提问策略

```
┌──────────────────────────────────────────────────┐
│           权衡问题提问策略                        │
│                                                  │
│  第一轮：方案展示                                 │
│  - 展示各方案的优缺点                             │
│  - 提供评估维度                                   │
│                                                  │
│  第二轮：优先级确认                               │
│  - 确认用户更看重哪个维度                         │
│  - 识别用户的决策标准                             │
│                                                  │
│  第三轮：风险评估                                 │
│  - 评估各方案的风险                               │
│  - 确认用户的风险承受能力                         │
│                                                  │
│  第四轮：最终决策                                 │
│  - 提供推荐方案                                   │
│  - 确认用户的选择                                 │
└──────────────────────────────────────────────────┘
```

## 评估维度

| 维度 | 说明 | 评估标准 |
|------|------|----------|
| 技术栈 | 使用的技术和工具 | 成熟度、社区支持、学习曲线 |
| 开发复杂度 | 实现难度 | 代码量、依赖数量、集成难度 |
| 性能 | 运行效率 | 响应时间、资源占用、并发能力 |
| 可扩展性 | 未来扩展能力 | 模块化、插件化、配置化 |
| 维护成本 | 长期维护难度 | 代码可读性、测试覆盖率、文档完整性 |
| 学习曲线 | 团队上手难度 | 文档质量、社区活跃度、示例数量 |

## 示例对话

```
用户：我想给 CodeY 添加用户认证功能

Agent：基于研究结果，我提供以下三个方案：

## 方案对比

| 维度 | 方案 A: JWT + Refresh Token | 方案 B: JWT 无 Refresh | 方案 C: Session |
|------|---------------------------|----------------------|-----------------|
| 技术栈 | JWT + Redis | JWT | Session + Cookie |
| 开发复杂度 | 中 | 低 | 低 |
| 安全性 | 高 | 中 | 高 |
| 可扩展性 | 好 | 好 | 差 |
| 维护成本 | 中 | 低 | 低 |

### 方案 A: JWT + Refresh Token
**优点**：
- 安全性最高（短 Token + Refresh Token）
- 适合分布式架构
- 用户体验好（无频繁登录）

**缺点**：
- 需要 Redis 存储 Refresh Token
- 开发复杂度中等

**适用场景**：分布式架构、高安全要求

### 方案 B: JWT 无 Refresh
**优点**：
- 实现简单
- 无状态

**缺点**：
- 安全性较低
- 用户体验差（频繁登录）

**适用场景**：简单应用、低安全要求

### 方案 C: Session
**优点**：
- 实现简单
- 安全性高

**缺点**：
- 有状态，不适合分布式
- 可扩展性差

**适用场景**：单体应用、传统架构

## 推荐方案

**推荐方案 A**，理由：
1. 安全性最高（短 Token + Refresh Token）
2. 适合分布式架构
3. 用户体验好（无频繁登录）

**你倾向于哪个方案？**

用户：选择方案 A
```

## 注意事项

1. **客观中立**：不要预设立场，让用户做出选择
2. **全面对比**：覆盖所有重要维度，不遗漏关键因素
3. **风险提示**：明确指出各方案的风险和限制
4. **推荐理由**：提供推荐方案时，给出充分的理由
