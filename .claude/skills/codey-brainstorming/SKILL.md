# 头脑风暴 Skill

结构化的头脑风暴流程，用于需求收集、研究调研和 Spec 生成。

## 何时激活

- 收到新的功能需求，需要澄清和细化
- 用户有一个模糊的想法，需要转化为可执行方案
- 在多种技术方案之间做出选择
- 生成 Spec 文档（design.md, api.md, test.md）
- 在动手写代码之前进行充分的调研和规划

## 核心流程

```
Phase 1: 需求收集 → Phase 2: 研究调研 → Phase 3: 方案生成 → Phase 4: 细节确认 → Phase 5: Spec 生成
```

### Phase 1: 需求收集

通过提问澄清：
- 核心目标和使用场景
- 约束条件和边界
- 优先级和 MVP 范围

### Phase 2: 研究调研

研究工具优先级：
1. **GitHub 代码搜索** - 查找类似实现、参考项目
2. **Context7 文档查询** - 查询库/框架官方文档
3. **Exa 网络搜索** - 最佳实践、技术文章

### Phase 3: 方案生成

提供 2-3 个可行方案，对比分析：

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 复杂度 | 低/中/高 | 低/中/高 | 低/中/高 |
| 性能 | 好/中/差 | 好/中/差 | 好/中/差 |
| 可扩展性 | 好/中/差 | 好/中/差 | 好/中/差 |

### Phase 4: 细节确认

确认技术细节、接口设计、测试策略。

### Phase 5: Spec 生成

在 `docs/specs/YYYY-MM-DD-<feature>/` 生成：
- `design.md` - 架构设计、数据模型、技术选型
- `api.md` - API 端点、请求/响应格式、错误码
- `test.md` - 测试策略、用例清单、覆盖率目标

## 核心原则

| 原则 | 说明 |
|------|------|
| 先理解后实现 | 充分理解需求后再动手 |
| 先提问后假设 | 有疑问就提问，不做假设 |
| 先 Spec 后代码 | Spec 确认后才开始写代码 |
| 多方案对比 | 每个决策点提供 2-3 个方案 |
| 用户决策 | Agent 提供分析，用户做出选择 |

## 集成

- **输出**：Spec 文档输入到 `codey-dev-workflow`
- **研究**：使用 GitHub、Context7、Exa 进行调研
