# Phase 1: 需求收集（Requirement Gathering）

## 目标

理解用户的核心需求，识别模糊点和约束条件。

## 流程

```
1. 用户描述需求（可能很模糊）
2. Agent 分析需求，识别模糊点
3. 向用户提出澄清问题
4. 收集约束条件（技术、时间、资源）
5. 识别用户的知识盲区
6. 整理需求文档（内部使用，不输出）
```

```

```

## 问题类型

需求收集阶段使用三种问题类型：

### 1. Clarifying Questions（澄清问题）

用于理解需求的本质和边界。

详见：[../questions/clarifying.md](../questions/clarifying.md)

**示例**：
- "这个功能的主要目的是什么？"
- "用户使用这个功能的典型场景是什么？"
- "有哪些约束条件？"
- "优先级如何？MVP 包含哪些内容？"
- "与现有功能的关系是什么？"
- "有参考实现或对标产品吗？"

### 2. Technical Questions（技术问题）

用于明确技术要求和限制。

详见：[../questions/technical.md](../questions/technical.md)

**示例**：
- "你倾向于哪种技术方案？"
- "有哪些性能要求？（响应时间、并发量）"
- "有哪些安全要求？（认证、加密、权限）"
- "需要支持哪些平台/浏览器？"
- "有哪些兼容性要求？"
- "数据量预估是多少？"

### 3. Trade-off Questions（权衡问题）

当存在多种方案时，帮助用户做出选择。

详见：[../questions/tradeoff.md](../questions/tradeoff.md)

**示例**：
- "方案 A 优点是...缺点是..."
- "方案 B 优点是...缺点是..."
- "你更看重哪个方面？（性能 vs 开发速度）"
- "可以接受的复杂度上限是什么？"

## 输出

需求收集阶段的输出是内部需求文档，包含：

```markdown
## 需求摘要

### 核心需求
- 需求描述
- 目标用户
- 使用场景

### 约束条件
- 技术约束
- 时间约束
- 资源约束

### MVP 范围
- 必须包含的功能
- 可以后续添加的功能

### 参考实现
- 对标产品
- 参考项目
```
