# Phase 3: 测试编写

## 概述

测试子 Agent 编写全面的测试用例，验证功能正确性。测试 Agent 独立于开发 Agent，确保测试的客观性。

## 负责人

测试子 Agent（独立于开发 Agent）

## 目标

编写全面的测试用例，验证功能正确性。

## 流程

1. 读取 Spec 文档（test.md, api.md）
2. 读取开发 Agent 的实现代码
3. 编写测试用例：
   - 单元测试：覆盖核心函数
   - 集成测试：覆盖 API 端点和模块交互
   - 边界测试：覆盖异常输入和边界条件
4. 运行测试
5. 报告结果：
   - 通过/失败统计
   - 失败详情
   - 覆盖率报告
6. 如有失败，反馈给开发 Agent

## 测试要求

| 测试类型 | 覆盖率目标 | 说明 |
|----------|-----------|------|
| Unit Tests | >= 80% | 单个函数/模块 |
| Integration Tests | >= 70% | 模块间交互 |
| Edge Cases | 100% 关键路径 | 边界值、异常输入 |

## 测试策略

### 单元测试

- 测试单个函数或模块
- Mock 外部依赖
- 覆盖正常路径和异常路径
- 测试边界条件

### 集成测试

- 测试模块间交互
- 测试 API 端点
- 测试数据库操作
- 测试外部服务集成

### 边界测试

- 空输入
- 超大输入
- 特殊字符
- 并发请求
- 超时场景

## 测试编写规范

### 测试命名

```javascript
describe('功能模块', () => {
  describe('具体功能', () => {
    it('应该在正常输入时返回正确结果', () => {
      // 测试代码
    });

    it('应该在空输入时抛出错误', () => {
      // 测试代码
    });
  });
});
```

### 测试结构

- **Arrange**：准备测试数据
- **Act**：执行被测试的操作
- **Assert**：验证结果

### Mock 策略

- 外部 API 调用必须 Mock
- 数据库操作使用内存数据库或 Mock
- 文件系统操作使用 Mock
- 时间相关操作使用 Mock

## 输出

- 测试代码文件
- 测试运行报告
- 覆盖率报告
- 失败项清单（如有）

## 报告格式

```markdown
## 测试报告

### 概览
- 总测试数：XX
- 通过：XX
- 失败：XX
- 跳过：XX
- 覆盖率：XX%

### 失败详情
（如有失败项）

### 覆盖率详情
- 文件 A：XX%
- 文件 B：XX%
```
