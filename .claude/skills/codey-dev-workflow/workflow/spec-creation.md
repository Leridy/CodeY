# Spec 创建

## 概述

Spec 是开发工作的契约文档，定义了功能的设计、API 接口和测试策略。所有开发工作必须基于 Spec 进行。

## Spec 文件结构

```
docs/specs/YYYY-MM-DD-<feature>/
├── design.md          # 设计文档
├── api.md             # API 规范
└── test.md            # 测试规范
```

## design.md 内容

```markdown
# <功能名称>

> 版本：v1.0.0
> 日期：YYYY-MM-DD
> 作者：<作者>
> 状态：[draft | review | approved]

## 概述

功能的简要描述和核心目标。

## 架构设计

### 整体架构

系统架构图和模块划分。

### 数据模型

核心数据结构定义。

### 技术选型

选用的技术栈和理由。

## 实现方案

### 模块 A

详细实现方案。

### 模块 B

详细实现方案。

## 依赖关系

与现有功能的依赖。

## 风险和约束

已知风险和应对方案。
```

## api.md 内容

```markdown
# <功能名称> API 规范

> 版本：v1.0.0
> 日期：YYYY-MM-DD

## 接口列表

### POST /api/<resource>

**描述**：创建资源

**请求体**：
```json
{
  "field": "value"
}
```

**响应**：
```json
{
  "success": true,
  "data": { ... }
}
```

**错误码**：

| 错误码 | 说明 |
|--------|------|
| 400 | 参数错误 |
| 401 | 未授权 |
| 500 | 服务器错误 |
```

## test.md 内容

```markdown

> 版本：v1.0.0
> 日期：YYYY-MM-DD

## 测试策略

### 单元测试

- 覆盖率目标：>= 80%
- 重点测试：核心函数、边界条件

### 集成测试

- 覆盖率目标：>= 70%
- 重点测试：API 端点、模块交互

### E2E 测试

- 重点测试：关键用户流程

## 测试用例清单

### 单元测试

- [ ] 测试用例 1：描述
- [ ] 测试用例 2：描述

### 集成测试

- [ ] 测试用例 1：描述
- [ ] 测试用例 2：描述

## 边界条件

- 空输入

- 并发请求

## 命名规范

- 目录名：`YYYY-MM-DD-<feature>`，如 `2026-07-05-user-auth`
- 特性名使用小写连字符：`user-auth`、`file-upload`、`agent-protocol`
- 版本号：`vX.Y.Z`（MAJOR.MINOR.PATCH）

## 版本管理规则

- **MAJOR**：接口不兼容、架构调整
- **MINOR**：功能补充
- **PATCH**：文档修正

## Spec 状态流转

```
draft → review → approved
```

- **draft**：初始草案，可自由修改
- **review**：提交审查，等待审批
- **approved**：已批准，进入开发阶段
