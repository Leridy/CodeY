# Phase 4: 代码审查

## 概述

审查子 Agent 确保代码质量、安全性和可维护性。审查 Agent 独立于开发和测试 Agent，确保客观性。

## 负责人

审查子 Agent（独立于开发和测试 Agent）

## 目标

确保代码质量、安全性和可维护性。

## 流程

1. 读取实现代码
2. 读取 Spec 文档（对照实现是否符合规范）
3. 执行审查：
   - 代码质量审查
   - 安全审查
   - 性能审查
4. 生成审查报告
5. 标记问题严重性

## 审查维度

| 维度 | 检查项 | 严重性 |
|------|--------|--------|
| 代码质量 | 可读性、命名、注释、结构 | MEDIUM |
| 安全性 | 硬编码密钥、注入风险、权限检查 | CRITICAL |
| 性能 | 算法复杂度、内存使用、N+1 查询 | HIGH |
| 可维护性 | 模块化、耦合度、测试覆盖 | MEDIUM |
| 规范符合性 | 是否符合 Spec 定义 | HIGH |

## 问题严重性

- **CRITICAL**：必须修复，阻塞提交
- **HIGH**：强烈建议修复
- **MEDIUM**：建议修复
- **LOW**：可选优化

## 审查检查清单

### 代码质量

- [ ] 代码可读性良好
- [ ] 命名清晰有意义
- [ ] 注释充分且准确
- [ ] 结构合理，模块化良好
- [ ] 无重复代码
- [ ] 函数长度合理（< 50 行）
- [ ] 文件长度合理（< 800 行）

### 安全性

- [ ] 无硬编码密钥或密码
- [ ] 输入验证完整
- [ ] SQL 注入防护
- [ ] XSS 防护
- [ ] CSRF 防护
- [ ] 权限检查正确
- [ ] 敏感数据加密
- [ ] 错误信息不泄露敏感数据

### 性能

- [ ] 算法复杂度合理
- [ ] 无 N+1 查询问题
- [ ] 内存使用合理
- [ ] 无不必要的循环
- [ ] 缓存使用得当

### 可维护性

- [ ] 模块化良好
- [ ] 耦合度低
- [ ] 测试覆盖充分
- [ ] 易于扩展
- [ ] 文档完整

### 规范符合性

- [ ] 实现符合 design.md 定义
- [ ] API 符合 api.md 定义
- [ ] 测试符合 test.md 定义

## 输出

- 审查报告（Markdown 格式）
- CRITICAL 和 HIGH 问题清单
- 改进建议列表

## 审查报告格式

```markdown
## 代码审查报告

### 概览
- 审查日期：YYYY-MM-DD
- 审查范围：<功能名称>
- 总体评价：[通过 | 需要修改 | 阻塞]

### 问题清单

#### CRITICAL
（无则留空）

#### HIGH
- 问题 1：描述
- 问题 2：描述

#### MEDIUM
- 问题 1：描述

#### LOW
- 问题 1：描述

### 改进建议
- 建议 1
- 建议 2

### 优点
- 优点 1
- 优点 2
```
