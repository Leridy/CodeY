# 检查冲突示例

## 场景

用户需要检查 `architecture.md` 文件是否存在冲突，确保文档的一致性。

## 步骤

### 1. 执行检查命令

```bash
doc-check-conflicts architecture.md
```

### 2. 扫描文件

Agent 扫描所有同名文件：

```
扫描文件...
  搜索模式：**/architecture.md
  搜索目录：docs/
  找到文件：
    1. docs/2026-07-01/architecture.md
    2. docs/2026-07-03/architecture.md
    3. docs/2026-07-05/architecture.md
```

### 3. 读取文件信息

Agent 读取每个文件的头部信息：

```
读取文件信息...

文件 1: docs/2026-07-01/architecture.md
  日期：2026-07-01
  阶段：架构设计
  版本：v1.0.0
  作者：CodeY Team
  大小：2.5 KB
  行数：85

文件 2: docs/2026-07-03/architecture.md
  日期：2026-07-03
  阶段：架构设计
  版本：v1.1.0
  作者：CodeY Team
  大小：3.2 KB
  行数：110

文件 3: docs/2026-07-05/architecture.md
  日期：2026-07-05
  阶段：架构设计
  版本：v1.3.0
  作者：CodeY Team
  大小：4.1 KB
  行数：145
```

### 4. 内容比较

Agent 比较文件内容差异：

```
比较内容...

比较 1: v1.0.0 vs v1.1.0
  差异行数：25
  差异章节：
    - 第 15 行：概述内容扩展
    - 第 28 行：架构图更新
    - 第 45 行：权限模型新增

比较 2: v1.1.0 vs v1.3.0
  差异行数：35
  差异章节：
    - 第 15 行：概述内容扩展
    - 第 28 行：架构图更新
    - 第 45 行：权限模型新增
    - 第 60 行：规则引擎新增
    - 第 75 行：部署架构更新

比较 3: v1.0.0 vs v1.3.0
  差异行数：60
  差异章节：
    - 第 15 行：概述内容扩展
    - 第 28 行：架构图更新
    - 第 45 行：权限模型新增
    - 第 60 行：规则引擎新增
    - 第 75 行：部署架构更新
    - 第 90 行：数据模型扩展
```

### 5. 生成冲突报告

Agent 生成冲突报告：

```
生成冲突报告...

冲突报告

文件名：architecture.md
位置：
  1. docs/2026-07-01/architecture.md (v1.0.0)
  2. docs/2026-07-03/architecture.md (v1.1.0)
  3. docs/2026-07-05/architecture.md (v1.3.0)

差异摘要：
  - v1.0.0 → v1.1.0：25 行差异
  - v1.1.0 → v1.3.0：35 行差异
  - v1.0.0 → v1.3.0：60 行差异

主要差异：
  1. 概述内容扩展（第 15 行）
  2. 架构图更新（第 28 行）
  3. 权限模型新增（第 45 行）
  4. 规则引擎新增（第 60 行）
  5. 部署架构更新（第 75 行）
  6. 数据模型扩展（第 90 行）

版本一致性：
  - v1.0.0：基础版本，内容完整
  - v1.1.0：中间版本，部分更新
  - v1.3.0：最新版本，内容最全

建议：
  1. 使用最新版本（v1.3.0）
  2. 删除旧版本（v1.0.0 和 v1.1.0）
  3. 确保所有引用指向最新版本

风险评估：
  - 内容丢失风险：低（最新版本包含所有内容）
  - 引用失效风险：中（可能有代码或文档引用旧版本）
  - 版本混乱风险：低（版本号清晰）
```

### 6. 提供解决方案

Agent 提供解决方案：

```
提供解决方案...

方案 A：使用最新版本
  操作：
    1. 保留 docs/2026-07-05/architecture.md (v1.3.0)
    2. 删除 docs/2026-07-01/architecture.md
    3. 删除 docs/2026-07-03/architecture.md
  优点：
    - 简单直接
    - 避免版本混乱
  缺点：
    - 可能丢失历史记录

方案 B：合并版本
  操作：
    1. 创建新版本 v1.4.0
    2. 合并所有版本的内容
    3. 保留合并历史
  优点：
    - 保留所有历史
    - 内容最完整
  缺点：
    - 操作复杂
    - 可能引入冲突

方案 C：保留所有版本
  操作：
    1. 保留所有版本
    2. 添加版本说明
    3. 更新引用指向
  优点：
    - 保留完整历史
    - 便于回溯
  缺点：
    - 占用空间
    - 可能造成混淆

推荐方案：A（使用最新版本）
  理由：
    1. 最新版本包含所有内容
    2. 操作简单，风险低
    3. 符合版本管理最佳实践
```

## 关键点

1. **全面扫描**：扫描所有同名文件，包括不同日期目录下的版本
2. **精确比较**：逐行比较内容差异，定位具体差异位置
3. **版本分析**：分析版本一致性，识别最新版本
4. **风险评估**：评估解决方案的风险等级
5. **多种方案**：提供多种解决方案，用户可根据实际情况选择
